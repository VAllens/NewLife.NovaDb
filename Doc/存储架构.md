# NewLife.NovaDb 存储架构文档

## 1. 概述

本文档描述 NovaDb 存储层的完整设计，涵盖文件组织、文件格式、页面管理、索引结构、WAL 日志、Binlog 日志以及并发控制等存储层面的实现细节。

### 1.1 设计目标

| 目标 | 说明 |
| :--- | :--- |
| **文件夹即数据库** | 每个数据库对应一个目录，拷贝目录即完成迁移/备份 |
| **MMF 驱动** | 以内存映射文件为主要存储介质，64 位系统虚拟地址空间 128TB，突破物理内存限制 |
| **冷热分离** | 热数据驻留内存提速，冷数据卸载至 MMF 省内存 |
| **每表独立文件组** | 降低锁粒度，表间互不影响 |
| **纯托管** | 无 Native 依赖，所有存储操作均为纯 C# 实现 |

### 1.2 存储层在整体架构中的位置

```
┌──────────────────────────────────────────────────────┐
│                  SQL / Engine / KV / Flux              │
├──────────────────────────────────────────────────────┤
│               事务层 (Tx) — MVCC                      │
├──────────────────────────────────────────────────────┤
│           WAL 日志层 (WAL + Binlog)                   │
│  WalWriter │ WalRecord │ WalRecovery                  │
│  BinlogWriter │ BinlogRecord                          │
├──────────────────────────────────────────────────────┤
│                 存储层 (Storage)                       │  ← 本文档范围
│  MmfPager │ PageCache │ FileHeader │ PageHeader        │
│  DatabaseDirectory │ TableFileManager                  │
├──────────────────────────────────────────────────────┤
│                 核心层 (Core)                          │
│  DbOptions │ DataType │ IDataCodec                     │
└──────────────────────────────────────────────────────┘
```

### 1.3 存储组织层级

```
数据库目录
  ├─ 数据库级文件（nova.db / nova.*.binlog）
  └─ 表级文件组（.nvd / .nvi / .wal / .lock）
       └─ 数据页（PageHeader + Data）
            └─ 行记录（RowLen + Columns）
```

## 2. 数据库目录结构

### 2.1 文件夹即数据库

每个数据库对应一个文件系统目录，目录下平铺所有文件。所有文件使用统一的表前缀，通过后缀区分类型。

### 2.2 文件分类

**数据库级文件**：

| 文件 | 命名规则 | 说明 |
| :--- | :--- | :--- |
| 元数据文件 | `nova.db` | 数据库完整元数据（表/列/索引定义），二进制格式 |
| Binlog 日志 | `nova.{SeqNo:D6}.binlog` | 主从同步/增量备份的逻辑日志，按文件滚动 |

**表级文件组**（每张表由以下文件组成）：

| 文件 | 后缀 | 说明 |
| :--- | :--- | :--- |
| 数据文件 | `.nvd` | 表的行数据，每表一个，通过 MMF 访问 |
| 主键索引 | `.nvi` | 主键索引文件，每表一个 |
| 二级索引 | `_{IndexName}.nvi` | 每个二级索引独立一个文件 |
| WAL 日志 | `.wal` | 写入时产生，checkpoint 后删除 |
| 写锁文件 | `.lock` | 嵌入模式下的表级写锁（服务器模式不需要） |

### 2.3 文件管理职责

*   **DatabaseDirectory**：负责数据库目录的创建、打开、枚举与删除。
*   **TableFileManager**：负责表级文件组的命名规范与路径生成。
*   **DatabaseManager**：在服务模式下管理多数据库实例与目录路由。

### 2.4 表文件命名规则

| 文件类型 | Nova Engine | Flux Engine (时序分片) |
| :--- | :--- | :--- |
| 数据文件 | `{TableName}.nvd` | `{TableName}.nvd`（当前分片）<br>`{TableName}_{yyyyMMdd}.nvd`（按天分片）<br>`{TableName}_{yyyyMMddHH}.nvd`（按小时分片） |
| 主键索引 | `{TableName}.nvi` | `{TableName}.nvi` |
| 二级索引 | `{TableName}_{IndexName}.nvi` | `{TableName}_{IndexName}.nvi` |
| WAL 文件 | `{TableName}.wal` | `{TableName}.wal` |
| 写锁文件 | `{TableName}.lock` | `{TableName}.lock` |

### 2.5 完整目录示例

```
StarDust/                              ← 文件夹即数据库

│   ── Binlog 日志 ──
```

## 3. 文件格式

### 3.1 文件头 (FileHeader)

所有 `.nvd`/`.nvi`/`.wal`/`.binlog` 和 `nova.db` 文件共享统一的 32 字节固定文件头：

```
偏移    大小    字段              说明
─────────────────────────────────────────────────────────
0       4B     Magic             魔数 0x4E4F5641 ("NOVA")
4       1B     Version           文件格式版本号（当前版本 1）
5       1B     FileType          文件类型（见下表）
6       1B     PageSizeShift     页大小位移（实际大小 = 1 << shift）
7       1B     Flags             特性标志（bit0=加密, bit1=压缩, bit2=只读）
8       8B     CreateTime        创建时间（UTC 毫秒）
16      4B     Checksum          CRC32 校验和（覆盖 bytes[0-15]）
20      12B    Reserved          预留扩展（填零）
─────────────────────────────────────────────────────────
合计    32B
```

**FileType 枚举**：

| 值 | 名称 | 说明 |
| :--- | :--- | :--- |
| 1 | Data | `.nvd` 数据文件 |
| 2 | Index | `.nvi` 索引文件 |
| 3 | Wal | `.wal` WAL 日志文件 |
| 4 | Binlog | `.binlog` 逻辑日志文件 |
| 5 | Metadata | `nova.db` 数据库元数据文件 |

**FileFlags 位标志**：

| 位 | 名称 | 说明 |
| :--- | :--- | :--- |
| bit0 | Encrypted | 文件已加密 |
| bit1 | Compressed | 文件已压缩 |
| bit2 | ReadOnly | 文件为只读 |

**校验和计算**：对 bytes[0-15]（Magic 到 CreateTime）执行 CRC32 计算，写入 bytes[16-19]。读取时重新计算并比对，不一致则文件损坏。

### 3.2 数据库元数据文件 (nova.db)

`nova.db` 是数据库的核心元数据文件，包含整个数据库的完整定义：

```
nova.db 结构：
├── FileHeader (32B)                     # 标准文件头，FileType=Metadata
├── DatabaseInfo                         # 数据库信息
│   ├── DatabaseName (string)            # 数据库名称
│   ├── CreateTime (datetime)            # 创建时间
│   └── ModifyTime (datetime)            # 最后修改时间
└── TableCatalog[]                       # 表目录（所有表的定义）
    ├── TableName (string)               # 表名
    ├── EngineType (byte)                # 引擎类型：0=Nova, 1=Flux, 2=KV
    ├── Comment (string)                 # 表注释
    ├── CreateTime (datetime)            # 建表时间
    ├── ColumnCount (int)                # 列数
    ├── Columns[]                        # 列定义列表
    │   ├── Name (string)                # 列名
    │   ├── DataType (byte)              # 数据类型
    │   ├── Length (int)                  # 长度（STRING/BINARY 类型）
    │   ├── IsNullable (bool)            # 是否可空
    │   ├── IsPrimaryKey (bool)          # 是否主键
    │   ├── IsAutoIncrement (bool)       # 是否自增
    │   ├── DefaultValue (variant)       # 默认值
    │   └── Comment (string)             # 列注释
    ├── IndexCount (int)                 # 索引数
    └── Indexes[]                        # 索引定义列表
        ├── Name (string)                # 索引名
        ├── IsUnique (bool)              # 是否唯一
        └── ColumnNames[] (string)       # 包含的列名列表
```

**修改机制**：DDL 操作（CREATE/ALTER/DROP TABLE）时，先写临时文件 `nova.db.tmp`，完成后原子重命名为 `nova.db`，保证崩溃安全。

**启动加载**：引擎启动时一次性读取 `nova.db`，在内存中构建表 Schema 缓存。

## 4. 页面管理

### 4.1 页面布局

每个 `.nvd` 数据文件由文件头 + 连续数据页组成：

```
┌──────────────────────────────────┐
│        FileHeader (32B)           │  ← 文件头
├──────────────────────────────────┤
│        Page 0                     │  ← 偏移 = 32 + 0 × PageSize
│  ┌──────────────────────────────┐│
│  │    PageHeader (32B)          ││
│  ├──────────────────────────────┤│
│  │    Data (PageSize - 32B)     ││
│  └──────────────────────────────┘│
├──────────────────────────────────┤
│        Page 1                     │  ← 偏移 = 32 + 1 × PageSize
│  ┌──────────────────────────────┐│
│  │    PageHeader (32B)          ││
│  ├──────────────────────────────┤│
│  │    Data (PageSize - 32B)     ││
│  └──────────────────────────────┘│
├──────────────────────────────────┤
│        ...                        │
└──────────────────────────────────┘

页面偏移计算：offset = FileHeader(32B) + PageId × PageSize
```

### 4.2 页头 (PageHeader)

每个数据页以 32 字节页头开始：

```
偏移    大小    字段              说明
─────────────────────────────────────────────────────────
0       8B     PageId            页 ID（全局唯一）
8       1B     PageType          页类型（见下表）
9       1B     Flags             页级标志（bit0=已加密, bit1=已压缩, bit2=脏页）
10      2B     Reserved          预留
12      8B     LSN               日志序列号（用于 WAL 恢复比对）
20      4B     Checksum          CRC32 校验和
24      4B     DataLength        页内有效数据长度
28      4B     Reserved          预留扩展（填零）
─────────────────────────────────────────────────────────
合计    32B
```

**PageType 枚举**：

| 值 | 名称 | 说明 |
| :--- | :--- | :--- |
| 0 | Empty | 空页（已释放或未使用） |
| 1 | Data | 数据页（存储行数据） |
| 2 | Index | 索引页 |
| 3 | Directory | 目录页（稀疏索引锚点） |
| 4 | Metadata | 元数据页 |

**PageFlags 位标志**：

| 位 | 名称 | 说明 |
| :--- | :--- | :--- |
| bit0 | Encrypted | 页面已加密 |
| bit1 | Compressed | 页面已压缩 |
| bit2 | Dirty | 脏页标记（WAL 未刷盘） |

### 4.3 数据页内行存储

数据页内行数据按追加方式顺序排列：

```
┌─────────────────────────────────────────┐
│          PageHeader (32B)                │
├─────────────────────────────────────────┤
│  Row 0: [RowLen(4B)] [Col0] [Col1] ...  │
│  Row 1: [RowLen(4B)] [Col0] [Col1] ...  │
│  Row 2: [RowLen(4B)] [Col0] [Col1] ...  │
│  ...                                     │
│  (空闲空间)                               │
└─────────────────────────────────────────┘
```

每行以 4 字节行长度前缀开头，后跟各列的二进制编码数据。列编码规则由 `IDataCodec` 定义。

### 4.4 数据编码 (IDataCodec)

列数据按类型进行二进制编码：

| DataType | C# 类型 | 编码方式 | 存储大小 |
| :--- | :--- | :--- | :--- |
| Boolean | `Boolean` | 1 字节（0/1） | 1B |
| Int32 | `Int32` | 小端 4 字节 | 4B |
| Int64 | `Int64` | 小端 8 字节 | 8B |
| Double | `Double` | IEEE 754 双精度 | 8B |
| Decimal | `Decimal` | 128 位 (16 字节) | 16B |
| String | `String` | 4B 长度前缀 + UTF-8 编码字节 | 4B + 变长 |
| ByteArray | `Byte[]` | 4B 长度前缀 + 原始字节 | 4B + 变长 |
| DateTime | `DateTime` | 8 字节 Ticks（100 纳秒精度） | 8B |

**NULL 值处理**：长度前缀为 -1 (`0xFFFFFFFF`) 表示 NULL，不跟随数据。

## 5. MMF 分页访问器 (MmfPager)

### 5.1 设计原理

`MmfPager` 基于 `MemoryMappedFile` 提供文件级别的分页读写。利用操作系统的虚拟内存管理机制，64 位系统下虚拟地址空间高达 128TB，OS 仅将实际访问的页映射到物理内存，无需将整个文件加载到内存。

```
物理内存 (有限)                     虚拟地址空间 (128TB)
┌──────────┐                     ┌──────────────────┐
│  热点页面  │ ←──── 映射 ────── │  Page 0          │
│  Page 3   │                    │  Page 1          │
│  Page 7   │                    │  Page 2          │
│  Page 12  │                    │  Page 3 (热)     │ ← 实际在物理内存
│           │                    │  ...             │
│           │                    │  Page 7 (热)     │ ← 实际在物理内存
│           │                    │  ...             │
│           │                    │  Page 12 (热)    │ ← 实际在物理内存
│           │                    │  ...             │
│           │                    │  Page N          │ ← OS 按需 page fault 加载
└──────────┘                     └──────────────────┘
                                          ?
                                 ┌──────────────────┐
                                 │  磁盘 .nvd 文件   │
                                 └──────────────────┘
```

### 5.2 核心能力

- **自动写入/验证 FileHeader**：新文件创建时写入文件头，已有文件打开时验证
- **按 PageId 随机读写**：`ReadPage(pageId)` / `WritePage(pageId, data)`
- **可选 CRC32 校验和**：写入时计算 CRC32 并存入 PageHeader，读取时重新计算验证
- **文件按需扩展**：写入超出当前文件范围的页时，自动扩展文件大小
- **线程安全**：内部使用锁保护文件流和 MMF 映射操作

### 5.3 页面偏移计算

```
页面偏移 = FileHeader(32B) + PageId × PageSize
```

默认 PageSize = 4096 字节，每页可用数据空间 = 4096 - 32（PageHeader）= 4064 字节。

### 5.4 配置项

| 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- |
| PageSize | 4096 | 页面大小（字节），必须为正整数 |
| EnableChecksum | true | 是否启用页面 CRC32 校验和验证 |
| PageCacheSize | 1024 | 页面缓存最大页数（LRU 策略） |

## 6. 页面缓存 (PageCache)

### 6.1 LRU 策略

`PageCache` 基于 `LinkedList<UInt64> + Dictionary<UInt64, CacheEntry>` 实现 O(1) 的 LRU 缓存：

```
最近访问 ←───────────────────────────→ 最久未访问
 [Page7] ←→ [Page3] ←→ [Page12] ←→ [Page5]
   ↑                                    ↑
   │                                    │
 命中时移到这里                        容量满时从这里淘汰
```

- **TryGet**：命中时将页移至链表头部（最近访问），返回缓存数据
- **Put**：容量满时淘汰链表尾部（最久未访问），再插入新页到头部
- **线程安全**：所有操作均在锁保护下执行

### 6.2 监控指标

| 指标 | 说明 |
| :--- | :--- |
| HitCount | 缓存命中次数 |
| MissCount | 缓存未命中次数 |
| Count | 当前缓存页数 |
| Capacity | 最大缓存页数 |

## 7. 索引结构

### 7.1 跳表 (SkipList)

NovaDb 使用跳表作为核心内存索引结构。跳表是一种概率性数据结构，通过多层有序链表实现 O(log N) 的查找/插入/删除。

```
Level 3:  HEAD ─────────────────────────── 50 ──────────────────── NIL
Level 2:  HEAD ────── 10 ──────────────── 50 ────── 70 ────────── NIL
Level 1:  HEAD ── 5 ─ 10 ── 20 ───────── 50 ── 60 ─ 70 ── 90 ── NIL
Level 0:  HEAD ── 5 ─ 10 ── 20 ── 30 ── 50 ── 60 ─ 70 ── 90 ── NIL
```

**实现参数**：

| 参数 | 值 | 说明 |
| :--- | :--- | :--- |
| MaxLevel | 16 | 最大层数，支持 2^16 = 65536 个节点的高效查找 |
| P | 0.5 | 升层概率，每个节点有 50% 概率出现在上层 |

**节点内容**：跳表节点存储索引键与行定位信息（`PageId + Offset`），多层指针用于快速查找与范围扫描。

**复杂度**：

| 操作 | 平均 | 最坏 |
| :--- | :--- | :--- |
| 查找 | O(log N) | O(N) |
| 插入 | O(log N) | O(N) |
| 删除 | O(log N) | O(N) |
| 范围扫描 | O(log N + K) | O(N + K) |

**并发控制**：所有跳表操作在 `_lock` 保护下执行，保证线程安全。

### 7.2 索引冷热分离

索引采用动态冷热分离策略，平衡内存使用与访问性能：

```
┌───────────────────────────────────────────┐
│     热索引 (HotIndexManager)               │
│     完整 SkipList 节点                      │
│     最近访问的数据段，内存中完整索引          │
├───────────────────────────────────────────┤
│     冷索引 (ColdIndexDirectory)             │
│     稀疏目录（每 N 行一个锚点）              │
│     长期未访问的数据段，仅保留锚点映射        │
└───────────────────────────────────────────┘
```

#### 7.2.1 热索引管理器 (HotIndexManager)

管理热段索引，将最近访问的数据段完整加载到内存跳表中：

- **热段配置**：
  - 热数据窗口：默认 600 秒（10 分钟），窗口内的数据保持热索引
  - 冷淘汰阈值：默认 1800 秒（30 分钟），超过则卸载为冷索引
  - 热段最大行数：默认 100 万行
  - 热度检查间隔：默认 60 秒

- **LRU 淘汰**：按最后访问时间排序，超过冷淘汰阈值的热段自动降级为冷索引
- **段元数据 (IndexSegment)**：记录段 ID、是否为热段、键范围（MinKey/MaxKey）、行数、最后访问时间

#### 7.2.2 冷索引目录 (ColdIndexDirectory)

长期未访问的数据段，仅在内存中保留稀疏目录，减少内存占用：

```
冷索引稀疏目录示例（每 1000 行一个锚点）：

锚点 0:  Key=1      → PageId=0,  Offset=0
锚点 1:  Key=1001   → PageId=5,  Offset=128
锚点 2:  Key=2001   → PageId=10, Offset=256
锚点 3:  Key=3001   → PageId=15, Offset=384
...
```

- **锚点间隔**：默认每 1000 行一个锚点
- **查找流程**：二分搜索稀疏目录定位到最近锚点 → 从锚点对应的页开始顺序扫描 → 找到目标行
- **访问提升**：频繁访问的冷段可自动提升为热段

#### 7.2.3 内存效果

| 场景 | 内存占用 |
| :--- | :--- |
| 1000 万行表，仅查最新 1 万行 | < 20MB（仅热段 + 冷段稀疏目录） |
| 1000 万行表，全表随机查 | 依赖 MMF page fault，性能下降但可用 |

### 7.3 索引文件 (.nvi)

#### 7.3.1 多索引多文件

一张表每个索引（主键/二级索引/唯一索引等）对应独立索引文件，便于独立加载、淘汰与重建：

- 主键索引：`{TableName}.nvi`
- 二级索引：`{TableName}_{IndexName}.nvi`

#### 7.3.2 nvi 文件结构

`.nvi` 文件存储可持久化的"目录 + 块/页"结构，**不是内存跳表节点的镜像**。内存 SkipList 用作热缓存结构，可从 `.nvi` 按需重建热点区域。

```
┌──────────────────────────┐
│  FileHeader (32B)         │  ← FileType=Index
├──────────────────────────┤
│  目录区域                  │
│  稀疏目录锚点 → 块偏移     │  ← 用于按需 MMF 映射与快速定位
├──────────────────────────┤
│  索引页序列                │
│  Page 0: 索引条目          │
│  Page 1: 索引条目          │
│  ...                      │
│  Page N: 最后一页           │
└──────────────────────────┘
```

**加载策略**：
- 引擎启动时，仅读取 `.nvi` 的稀疏目录锚点到内存（冷索引）
- 首次查询某范围时，从 `.nvi` 对应块加载到内存跳表（热索引）
- LRU 淘汰时，热索引卸载回冷索引稀疏目录

## 8. 数据文件 (.nvd)

### 8.1 Nova Engine 数据文件

Nova Engine 每张表一个 `.nvd` 文件，逻辑上不设行数上限，数据增长主要消耗磁盘资源。

```
{TableName}.nvd 结构：
┌──────────────────────────┐
│  FileHeader (32B)         │  ← FileType=Data
├──────────────────────────┤
│  数据页序列                │
│  Page 0: PageHeader + 行数据  │
│  Page 1: PageHeader + 行数据  │
│  ...                      │
│  Page N: 最后一页           │
└──────────────────────────┘
```

- **不分片**：单表单文件，10G、100G 的数据文件在 MMF 下没有问题，OS 只映射实际访问的页，物理内存按需分配
- **如需逻辑分表**（按时间/Hash），由 XCode ORM 层在应用层面完成

### 8.2 Flux Engine 时序分片

Flux Engine 按时间分片，每个分片一个 `.nvd` 文件：

```
AppLog.nvd                    ← 当前活跃分片（追加写入）
AppLog_20250101.nvd           ← 2025-01-01 的历史分片
AppLog_20250102.nvd           ← 2025-01-02 的历史分片
AppLog_20250103.nvd           ← 2025-01-03 的历史分片
```

- **分片粒度**：按配置决定，支持按天（`yyyyMMdd`）或按小时（`yyyyMMddHH`）
- **追加写入**：时序数据仅追加（Append Only），不更新历史分片
- **TTL 保留**：支持按时间/文件大小自动删除旧分片

### 8.3 KV 表数据文件

KV 表复用 Nova Engine 的 `.nvd` 文件，使用固定 Schema：

```
固定 Schema: Key (STRING(200), PK) | Value (BLOB) | TTL (DATETIME, Nullable)
```

每个 KV 表独立文件组，可配置不同的默认 TTL。

## 9. WAL 日志

### 9.1 设计目标

WAL (Write-Ahead Logging) 确保崩溃安全：事务修改先写入 WAL 日志，再更新数据文件。崩溃后可通过 WAL 重放恢复到一致状态。

### 9.2 WAL 模式

| 模式 | 行为 | 适用场景 |
| :--- | :--- | :--- |
| Full | 每次写入同步刷盘（`Flush`） | 金融、关键数据 |
| Normal | 异步刷盘（1 秒间隔） | 通用场景（默认） |
| None | 不写 WAL | 测试、只读场景 |

### 9.3 WAL 记录格式

每条 WAL 记录固定 37 字节头部 + 可变数据：

```
偏移    大小    字段              说明
─────────────────────────────────────────────────────────
0       8B     LSN               日志序列号（单调递增）
8       8B     TxId              事务 ID
16      1B     RecordType        记录类型（见下表）
17      8B     PageId            页 ID（仅 UpdatePage 有效）
25      4B     DataLength        数据长度
29      8B     Timestamp         时间戳
─────────────────────────────────────────────────────────
37      变长    Data              页数据（仅 UpdatePage）
```

**RecordType 枚举**：

| 值 | 名称 | 说明 |
| :--- | :--- | :--- |
| 1 | BeginTx | 开始事务 |
| 2 | UpdatePage | 更新页面（携带完整页数据） |
| 3 | CommitTx | 提交事务 |
| 4 | AbortTx | 回滚事务 |
| 5 | Checkpoint | 检查点标记 |

### 9.4 WAL 生命周期

```
事务写入 ──→ WAL 追加记录 ──→ Commit ──→ 后台 Checkpoint ──→ 删除 .wal
                                              │
                                              ↓
                                    脏页刷新到 .nvd 文件
```

| 阶段 | 行为 |
| :--- | :--- |
| **写入** | 事务写入时，WAL 记录追加到 `{TableName}.wal` |
| **Checkpoint** | 后台线程将 WAL 中的脏页刷新到 `.nvd` 文件 |
| **清理** | Checkpoint 完成后，`.wal` 文件删除（或截断为 0） |
| **崩溃恢复** | 启动时发现 `.wal` 存在 → 执行 WalRecovery 重放已提交事务 → 恢复后删除 |
| **只读模式** | 不创建 `.wal` 文件，跳过 WAL 初始化 |

### 9.5 崩溃恢复 (WalRecovery)

两阶段恢复策略：

1. **分析阶段 (Analysis)**：顺序扫描 WAL，收集所有 `CommitTx` 记录的事务 ID，确定已提交事务集合
2. **重做阶段 (Redo)**：对已提交事务的 `UpdatePage` 记录，调用 `applyPageUpdate(pageId, data)` 重放页面更新

```
WAL 文件 ──→ 扫描 CommitTx ──→ 过滤已提交事务的 UpdatePage ──→ 重放到 .nvd ──→ 删除 .wal
```

**安全性**：未提交事务的 `UpdatePage` 记录不会被重放，保证崩溃后数据一致性。

## 10. Binlog 日志

### 10.1 WAL 与 Binlog 的区别

| 维度 | WAL | Binlog |
| :--- | :--- | :--- |
| **层级** | 表级（每表一个 `.wal`） | 数据库级（全库共享 `.binlog`） |
| **记录内容** | 物理页面变更（UpdatePage） | 逻辑操作（INSERT/UPDATE/DELETE/DDL） |
| **用途** | 崩溃恢复 | 主从复制 + 增量备份 + 审计 |
| **生命周期** | 短暂（checkpoint 后删除） | 持久保留（按策略滚动清理） |
| **写入时机** | 事务写入时（提交前） | 事务提交后 |

两者职责正交、互不依赖。WAL checkpoint 后删除不影响 Binlog；Binlog 手工删除不影响 WAL 崩溃恢复。

### 10.2 Binlog 记录格式

```
BinlogRecord:
├── LSN (Int64)              # 全局单调递增序号（断点续传标识）
├── Timestamp (Int64)        # 提交时间 (UTC 毫秒)
├── TableName (String)       # 目标表名
├── OpType (Byte)            # 操作类型：1=Insert, 2=Update, 3=Delete, 4=DDL
├── RowKey (Variant)         # 主键值（Update/Delete 时定位行）
├── PayloadLength (Int32)    # 数据负载长度
└── Payload (Byte[])         # Insert/Update 的行数据 / DDL 的 SQL 文本
```

### 10.3 文件管理

- **文件命名**：`nova.{SeqNo:D6}.binlog`（如 `nova.000001.binlog`、`nova.000002.binlog`）
- **滚动机制**：单文件达到大小阈值（默认 256MB，`BinlogMaxFileSize` 配置）时关闭当前文件，创建下一个序号文件
- **序号永远递增**：不回绕、不重用，保证全局唯一
- **文件内索引**：FileHeader 之后记录 LSN 范围（起始 LSN ~ 结束 LSN），可快速定位 LSN 所在文件

### 10.4 自动清理策略

后台清理线程定期检查，三种策略可叠加使用（任一条件满足即清理）：

| 策略 | 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| 从节点确认 | 自动 | — | 所有从节点已确认同步的文件可安全清理 |
| 按天数保留 | `BinlogRetentionDays` | 0（不限） | 超过保留天数的旧文件自动删除 |
| 按总大小限制 | `BinlogMaxTotalSize` | 0（不限） | 总大小超过上限时从最早文件开始删除 |

**清理安全性**：
- 当前正在写入的活跃文件永远不会被清理
- 按天数/总大小清理时不检查从节点同步状态（运维主动决策，引擎尊重）

### 10.5 手工删除安全性

Binlog 文件设计为**可安全手工删除**的独立文件：

- 每个 `.binlog` 文件自包含（FileHeader + 完整记录序列），文件之间无交叉依赖
- 引擎启动时扫描数据库目录，按文件名排序重建 Binlog 文件列表，自动感知已删除的文件
- 直接 `rm` 旧文件不会导致引擎崩溃，唯一代价是未同步该文件的从节点需全量重建

## 11. 并发控制

### 11.1 并发模型

| 模式 | 引擎并发 | 跨进程锁 | 说明 |
| :--- | :--- | :--- | :--- |
| **嵌入模式** | MVCC 多读多写（进程内） | 表级文件锁（`{TableName}.lock`） | 进程内 MVCC 行级不阻塞；跨进程通过文件锁协调写入 |
| **服务器模式** | MVCC 多读多写 | 无（单进程） | 所有请求通过 SqlEngine 统一调度 |
| **只读模式** | 只读 | 无锁 | 不创建 `.wal` 和 `.lock`，多进程可同时只读打开 |

### 11.2 表级文件锁

嵌入模式下使用 `{TableName}.lock` 文件实现跨进程互斥：

- **锁文件**：0 字节文件，仅利用 `FileStream` 排他锁实现互斜
- **锁粒度**：每表独立，写 `User` 表不影响 `App` 表的读写
- **一写多读**：同一时刻只允许一个进程写入某表，但多个进程可同时读取

### 11.3 与 SQLite 对比

SQLite 使用库级锁，写入时锁定整个数据库文件，所有表读写都受影响。NovaDb 每表独立文件组让锁粒度缩小到单表，并发度显著提升。

## 13. CRUD 数据流（存储视角）

### 13.1 写入流程

```
Insert 请求
    │
    ├─→ 序列化行数据（IDataCodec）
    │
    ├─→ WAL 追加 BeginTx + UpdatePage + CommitTx
    │     └─→ Full 模式：同步 Flush
    │     └─→ Normal 模式：异步 1s Flush
    │
    ├─→ 内存 SkipList 插入新条目
    │
    ├─→ 数据页写入（MmfPager.WritePage）
    │     └─→ PageCache 更新
    │
    └─→ Binlog 记录（服务器模式，事务提交后）
```

### 13.2 读取流程

```
Get 请求 (主键查找)
    │
    ├─→ 热索引查找（HotIndexManager / SkipList）
    │     └─→ 命中：直接返回数据偏移
    │
    ├─→ 未命中：冷索引查找（ColdIndexDirectory）
    │     └─→ 二分搜索稀疏目录 → 定位锚点
    │     └─→ 从 .nvi 加载对应索引块到热索引
    │     └─→ 从 .nvd 按 MMF 映射读取数据页
    │
    ├─→ PageCache 检查
    │     └─→ 命中：返回缓存页
    │     └─→ 未命中：MmfPager.ReadPage → OS page fault → 磁盘读取
    │
    ├─→ MVCC 可见性判断
    │
    └─→ 反序列化行数据（IDataCodec）
```

### 13.3 Checkpoint 流程

```
后台 Checkpoint 线程
    │
    ├─→ 扫描 WAL 中的 UpdatePage 记录
    │
    ├─→ 将脏页刷新到 .nvd 文件
    │     └─→ MmfPager.WritePage
    │     └─→ PageCache 更新
    │
    └─→ 删除（或截断）.wal 文件
```

## 14. 设计决策记录

| # | 决策 | 要点 |
| :--- | :--- | :--- |
| D1 | **纯托管实现** | 不引入 native 依赖，保持跨平台一致性 |
| D2 | **MMF 作为主要存储** | 利用操作系统虚拟内存管理，64 位系统虚拟地址空间 128TB |
| D3 | **SkipList 而非 B+ 树** | 实现简单、并发友好，适合内存索引场景 |
| D4 | **表级 WAL** | 每表独立 WAL，减少并发竞争，简化恢复流程 |
| D5 | **冷热分离索引** | 热段内存索引 + 冷段稀疏目录，千万行仅查热点时内存 < 20MB |
| D6 | **Nova Engine 不分片** | 单表单文件，MMF 可高效处理超大文件，分片路由复杂度大于收益 |
| D7 | **nova.db 二进制元数据** | 取代系统表概念，DDL 原子重写保证崩溃安全 |
| D8 | **Binlog 独立于 WAL** | 职责正交互不依赖，WAL 崩溃恢复，Binlog 主从复制 |
| D9 | **Binlog 可安全手工删除** | 文件自包含无交叉依赖，去掉索引文件依赖 |

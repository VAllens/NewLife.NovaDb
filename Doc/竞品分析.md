# NewLife.NovaDb 竞品分析（定位与取舍）

> 目标：帮助团队与用户回答“为什么选 NovaDb/不选 NovaDb”，并将对标结果映射到《需求规格说明书.md》的范围与非目标。

更新日期：2026年2月23日

---

## 1. 一句话定位

NovaDb 是一个 **纯 .NET 托管**、支持 **嵌入式/服务器双模** 的“混合型数据库内核”，将 **关系型（Nova）**、**时序+MQ（Flux）**、**KV** 放在同一套引擎与一致性/存储体系内，优先追求 **工程集成效率与低运维成本**，而不是追求 SQL 全功能或分布式强一致。

---

## 2. 需求范围对齐（我们对标的是什么）

NovaDb v1 的对外承诺以《需求规格说明书.md》为准，核心要点：

- **P0**：SQL 常用子集（DDL/DML/SELECT 基础）、Read Committed + 崩溃恢复、ADO.NET Provider、服务器模式 TCP/IPv6、EXPLAIN 与慢查询日志。
- **P1**：JOIN/子查询、Flux（时序+MQ）的增强能力、KV、主从复制、系统表/指标、物化视图。
- **明确非目标（v1）**：触发器/存储过程/普通视图/窗口函数/CTE 等高级 SQL；不做自动故障切换；不做分库分表与水平分片。

---

## 3. 竞品版图与“替代关系”

NovaDb 的典型替代关系不是“替代所有数据库”，而是针对特定场景减少组件数量：

- **嵌入式 .NET 应用**：替代 SQLite（能力/并发/容量受限的场景）。
- **小型服务/单体应用**：替代“单机 MySQL/PostgreSQL + Redis + MQ 的部分组合”，在可接受的功能子集下换取更低运维。
- **时序 + 队列一体化**：替代“Influx/TDengine + Redis Stream/Kafka”的轻量组合，强调一套存储、一套权限/运维（若 v1 暂无权限则为后续）。

### 4.2 NovaDb 的取舍

- 取：对“常用 SQL + ADO.NET + 工程集成”做强承诺，强调可验收、可诊断。
- 舍：不追求 SQL 全功能（窗口函数/CTE/触发器/存储过程等）。

## 5. 时序（InfluxDB / TDengine / TimescaleDB / QuestDB）对比

| 维度 | InfluxDB | TDengine | TimescaleDB | QuestDB | **NovaDb Flux** |
| --- | --- | --- | --- | --- | --- |
| **实现语言** | Go (v2) / Rust (v3) | C | C (PG 扩展) | Java | **C#** |
| **开源协议** | MIT (v2) / Apache 2.0 (v3) | AGPL v3 | Apache 2.0 (社区版) | Apache 2.0 | **MIT** |
| **部署方式** | 独立服务 | 独立服务/集群 | PG 扩展 | 独立服务 | **嵌入 / 独立服务** |
| **数据模型** | Tag + Field + Timestamp | 超级表 (STable) | 普通表 + 时间分区 | 表 + 时间列 | **时间分片表 (按天/小时)** |
| **写入模式** | Line Protocol (追加) | 追加写 | SQL INSERT (追加) | 追加写 | **追加写 (Append Only) + 分片** |
| **存储引擎** | TSM / IOx (列存储) | VNode 分片 | 基于 PG 堆表 + 分区 | 列存储 (零 GC) | **时间分片文件 + WAL + MMF** |
| **查询语言** | Flux / InfluxQL / SQL (v3) | 类 SQL (TQL) | 标准 SQL | 标准 SQL | **SQL (扩展语法)** |
| **降采样/聚合** | 连续查询 (Flux) | 内置 INTERVAL | TimescaleDB Toolkit | 内置采样 | **规划支持（连续查询、降采样）** |
| **保留策略 (TTL)** | ✅ (Retention Policy) | ✅ | ✅ | ✅ | **✅ (按时间/容量自动删除)** |
| **压缩** | Gorilla + RLE | 列压缩 | PG TOAST | 列压缩 | **✅（时间分片 + 冷热分离压缩）** |
| **连续聚合** | ✅ | ✅ | ✅ | ✅（实现差异） | **规划/逐步实现（对齐规格书）** |
| **时间窗口聚合** | ✅ (INTERVAL 子句) | ✅ (GROUP BY \<interval\>) | ✅ (GROUP BY time_bucket) | ✅ | **✅ (GROUP BY INTERVAL)** |
| **集群/分布式** | ✅ (企业版) | ✅ (原生) | ❌ (靠 PG 方案) | ✅ (企业版) | **✅（集群 + 一主多从 Binlog 同步）** |
| **MQ 融合** | ❌ | ❌ | ❌ | ❌ | **✅ (同引擎内置 Stream/MQ + 消费组)** |
| **消息顺序保证** | ❌ | ❌ | ❌ | ❌ | **✅ (全局有序，单 Stream)** |
| **消息可靠性** | ❌ | ❌ | ❌ | ❌ | **✅ (At-Least-Once + Pending + Ack)** |
| **运维复杂度** | 中 | 中偏高 | 低 (PG 生态) | 低 | **极低（文件夹即库）** |
| **适用规模** | 中大型 | 中大型 | 中大型 | 中大型 | **中大型/嵌入式** |
| **.NET 集成** | HTTP SDK | 原生驱动 | Npgsql | HTTP / PGWire | **原生嵌入 + ADO.NET** |

**NovaDb Flux Engine 特色说明**：

- **内置 MQ + 时序融合**：同一个引擎同时支持时序数据和消息队列，避免引入 Kafka/Redis Stream 等额外组件
- **严格顺序保证**：每个 Stream 单分区设计，保证全局消息顺序一致，消费组位点自动管理
- **消费者容灾**：支持 Pending（待确认）+ Ack（确认）机制，确保消息至少被消费一次
- **延迟队列与死信队列**：原生支持延迟投递、死信转移，无需外部配置
- **时序与 MQ 统一查询**：均支持标准 SQL 查询，包括时间窗口聚合、GROUP BY INTERVAL 等
- 以 `time_bucket` 等能力支撑时序聚合与物化视图（详见函数清单与规格书 §5.8）

| 维度 | Kafka | RabbitMQ | Redis Stream | RocketMQ | Pulsar | **NovaDb Flux MQ** |

| **消息 TTL/保留** | ✅ (按时间/大小) | ✅ (Per-Message TTL) | ✅ (MAXLEN/MINID) | ✅ (按时间) | ✅ (按时间/大小) | **✅ (按时间/大小自动删除)** |
| **SQL 查询消息** | ❌ (KSQL 独立组件) | ❌ | ❌ | ❌ | ✅ (Pulsar SQL) | **✅ (原生 SQL 查询)** |
| **吞吐量** | 极高 (百万/s) | 中 (万级/s) | 高 (十万/s) | 高 (十万/s) | 极高 (百万/s) | **中等（受 .NET GC 与单机限制）** |
| **集群** | ✅ (原生) | ✅ (镜像/仲裁队列) | ✅ (Redis Cluster) | ✅ (原生) | ✅ (原生) | **✅（集群 + 主从同步）** |
| **运维可观测** | Kafka Manager / JMX | Management UI | INFO / MONITOR | Dashboard | Admin API | **系统表 SQL 查询** |
| **适用规模** | 大型/超大型 | 大型/超大型 | 中大型 | 大型/超大型 | 大型/超大型 | **大型/嵌入式** |
| **.NET 集成** | Confluent.Kafka | RabbitMQ.Client | StackExchange.Redis | rocketmq-client-csharp | DotPulsar | **原生嵌入 + ADO.NET** |
| **适用场景** | 大数据管道/事件溯源 | 企业集成/任务分发 | 轻量级内部队列 | 电商/金融 | 多租户/云原生 | **嵌入式内部队列 / IoT 消息** |

NovaDb 适合“内部任务队列/事件流水/轻量消息”场景；不适合以 MQ 为核心的大规模多租户消息平台。

### 6.1 补充：大吞吐 MQ 的“为什么不选”

| 维度 | Kafka / Pulsar / RocketMQ 等 | NovaDb Flux MQ |
| --- | --- | --- |
| 核心目标 | 超高吞吐、多租户、大规模分区 | 低运维、同引擎内置、可 SQL 查询 |
| 典型依赖 | 独立集群与周边生态 | 可嵌入或单进程服务化 |
| 代价 | 部署/运维/生态复杂 | 吞吐上限受单机与 .NET 运行时影响 |

## 7. KV / NoSQL（Redis / RocksDB / MongoDB / etcd）对比

| 维度 | Redis | MongoDB | LevelDB / RocksDB | etcd | **NovaDb KV** |
| --- | --- | --- | --- | --- | --- |
| **实现语言** | C | C++ | C++ | Go | **C#** |
| **数据模型** | KV + 多种数据结构 | 文档 (BSON) | 有序 KV | KV (强一致) | **KV 表 (Key + Value + TTL)** |
| **部署方式** | 独立服务 / 集群 | 独立服务 / 副本集 | 嵌入式库 | 独立集群 (3/5 节点) | **嵌入 / 独立服务** |
| **持久化** | RDB / AOF (可选) | WiredTiger (默认持久) | WAL + SSTable | WAL + Snapshot | **WAL + MMF (强持久化)** |
| **事务** | MULTI/EXEC (弱事务) | 多文档事务 (4.0+) | 批量写 (原子) | Mini 事务 (Revision) | **MVCC 事务 (Read Committed)** |
| **查询能力** | 命令式 (无 SQL) | 丰富查询 + 聚合管道 | Get/Put/Scan | Range + Watch | **SQL 查询 + API** |
| **内存模型** | 全内存为主 | 工作集缓存 (WiredTiger) | 块缓存 (Block Cache) | 全内存 (BoltDB) | **冷热分离 (热段内存+冷段 MMF)** |
| **数据结构** | String/Hash/List/Set/ZSet/Stream | 文档/数组/嵌套 | 有序字节 KV | 扁平 KV | **Key + Value + TTL（每行）** |
| **TTL 支持** | ✅ (Key 级别) | ✅ (TTL 索引) | ❌ (需自实现) | ✅ (Lease) | **✅（每行 TTL，支持过期清理）** |
| **发布订阅** | ✅ (Pub/Sub + Stream) | ✅ (Change Stream) | ❌ | ✅ (Watch) | **✅ (Flux Engine MQ)** |
| **分布式** | Cluster / Sentinel | 副本集 / 分片 | ❌ (嵌入式) | Raft 共识 (强一致) | **✅（集群 + 一主多从）** |
| **运维复杂度** | 低 | 中 | 极低 (嵌入) | 中 | **极低（文件夹即库）** |
| **适用规模** | 大型/超大型 | 大型/超大型 | 嵌入式 | 大型 | **大型/嵌入式** |
| **适用场景** | 缓存/会话/排行榜/队列 | 文档存储/内容管理 | 嵌入式存储/LSM 底层 | 配置中心/服务发现 | **缓存/会话/配置/分布式锁** |
| **.NET 集成** | StackExchange.Redis | MongoDB.Driver | 需 P/Invoke | dotnet-etcd | **原生嵌入 + ADO.NET** |

**NovaDb KV 模式特色说明**：

- **多表隔离**：同一数据库支持多个独立的 KV 表，每表可配置不同默认 TTL
- **SQL + API 双访问**：既支持标准 SQL 查询，也提供 Get/Set/Add/Inc/Remove 等编程 API
- **冷热分离**：KV 数据也支持冷热分离，常访问的热键常驻内存，冷键自动转移到 MMF
- **TTL 与过期清理**：每行支持 TTL 设置，过期数据自动不可见，后台异步清理

### 7.1 补充：NoSQL/配置中心类竞品的定位说明

| 维度 | MongoDB | etcd | **NovaDb KV** |
| --- | --- | --- | --- |
| 核心模型 | 文档（JSON/BSON） | 强一致 KV（共识） | KV 表（Key + Value + TTL） |
| 强一致/共识 | 事务能力强但定位不同 | 强一致（Raft） | v1 侧重单机/主从复制，不对标共识型强一致 |
| 典型场景 | 内容/文档/灵活 Schema | 配置中心/服务发现 | 会话/缓存/配置/本地持久 KV（同引擎集成） |

| **定时修正** | ❌ (依赖 Merge) | ✅ (Compaction) | ✅ (Policy 调度) | ✅ (定时刷新) | **✅ (Cron 调度修正)** |
| **乱序数据处理** | ⚠️ (延迟处理窗口) | ✅ (Late Data 处理) | ✅ (修正窗口) | ✅ (延迟索引) | **✅ (修正窗口 + 延迟数据重算)** |
| **多粒度汇总** | ✅ (手动建多层 MV) | ✅ (Rollup 层次) | ✅ (不同 time_bucket) | ✅ (多 Transform) | **✅ (分层聚合：1分钟→1小时→1天)** |
| **聚合函数支持** | SUM/COUNT/AVG/MIN/MAX/...(60+) | SUM/COUNT/MIN/MAX/... | SUM/COUNT/AVG/MIN/MAX/... | SUM/COUNT/AVG/MIN/MAX/... | **SUM/COUNT/AVG/MIN/MAX (P1)** |
| **MQ 消费触发聚合** | ❌ (需 Kafka) | ✅ (Kafka 摄取) | ❌ (需外部 MQ) | ❌ (需 Logstash) | **✅ (内置 MQ + 实时聚合)** |
| **修正与源表对比** | ❌ (仅 Merge) | ✅ (Compaction 验证) | ✅ (定时重算) | ❌ (仅覆盖) | **✅ (GROUP BY 对比修正)** |
| **SQL 语法** | CREATE MATERIALIZED VIEW | Ingestion Spec (JSON) | CREATE CONTINUOUS AGGREGATE | Transform API (REST) | **CREATE MATERIALIZED VIEW ... REFRESH INCREMENTAL** |
| **系统表监控** | system.parts / merges | sys.segments | timescaledb_information.* | _cat/transforms | **_sys.materialized_views** |
| **外部组件依赖** | 可能需要 Kafka 等 | 通常需要 Kafka 等 | 依赖 PG 生态 | 依赖 ES 集群 | **尽量内聚（时序 + MQ + 聚合同引擎）** |
| **运维复杂度** | 中 (分布式表复杂) | 高 (组件多) | 低 (PG 生态) | 中 (集群配置) | **极低（文件夹即库）** |
| **适用场景** | 专业 OLAP / 复杂多维分析 | 大规模实时分析 | 时序 OLAP | 日志/检索分析 | **时序 OLTP + OLAP 融合** |
| **.NET 集成** | HTTP / Native Client | HTTP API | Npgsql | NEST | **原生嵌入 + ADO.NET** |

NovaDb 不以替代专业 OLAP 为目标；更强调“把常用聚合与预计算收进同一套引擎”。

**NovaDb 物化视图特色说明**：

1. **增量更新 + 定时修正混合架构**
   - **增量更新**：时序数据写入时，实时聚合引擎在内存中维护聚合状态，批量刷新到统计表（减少写放大）
   - **定时修正**：定时（如每15分钟）对源表执行 GROUP BY 聚合，与统计表对比修正，处理延迟数据和乱序问题
   - **优势**：兼顾实时性（秒级延迟）与准确性（定期修正），适合时序数据的"先快后准"场景

2. **MQ 消费自动聚合**
   - 时序数据通过 MQ 写入时，消费者可配置自动聚合逻辑
   - 在指定维度（如 `time_bucket` + `product_id`）上执行累加操作
   - 类似 Elasticsearch Transform，但无需外部 MQ（Logstash/Kafka）

3. **聚合状态存储（类似 ClickHouse）**
   - 使用 `-State` 聚合函数（`sumState`、`countState` 等）存储中间状态
   - 自动合并相同维度的状态，支持增量累加
   - 查询时使用 `-Merge` 函数（`sumMerge`、`avgMerge`）返回最终结果

4. **多粒度汇总（Rollup）**
   - 支持分层聚合：1分钟表 → 15分钟表 → 1小时表 → 1天表
   - 上层从下层聚合，减少计算量（避免重复扫描源表）
   - 历史数据自动压缩：仅保留高层次汇总，删除细粒度数据

5. **乱序数据处理**
   - 保留可调整的修正窗口（如最近2小时）
   - 窗口内的延迟数据触发增量更新
   - 超出窗口的数据触发全量修正（重新 GROUP BY）

**示例配置**：

```sql
CREATE MATERIALIZED VIEW order_stats_hourly 
REFRESH INCREMENTAL AS
SELECT 
    time_bucket('1 hour', order_time) as hour,
    product_id,
    SUM(amount) as total_amount,
    COUNT(*) as order_count
FROM orders
GROUP BY hour, product_id;

-- 配置刷新策略
ALTER MATERIALIZED VIEW order_stats_hourly 
SET refresh_interval = '10s',
    buffer_size = '10MB',
    reconciliation_schedule = '0 */15 * * * *',
    reconciliation_window = '1 hour';
```

**与竞品对比优势**：

- **vs ClickHouse**：NovaDb 支持定时修正（ClickHouse 仅依赖 Merge），更适合乱序数据场景
- **vs TimescaleDB**：NovaDb 支持增量更新（TimescaleDB 定时全量刷新），实时性更好
- **vs Druid**：NovaDb 无需外部 Kafka，内置 MQ + 聚合引擎，架构更简单
- **vs Elasticsearch**：NovaDb 支持 SQL 原生查询，无需 REST API 转换

### 11.2 决策流程

1. **首先问**："是 .NET 项目吗？"
   - 是 → 继续；否 → 考虑其他方案
2. **其次问**："能接受嵌入或单机服务化吗？"
   - 是 → 继续；否 → 需要成熟的分布式 RDBMS
3. **再问**："场景主要是 OLTP 还是 OLAP？"
   - OLTP → 适合；OLAP → 需要结合物化视图或外接专业库
4. **最后问**："需要高级 SQL 特性吗？"
   - 是 → PostgreSQL/SQL Server 更合适；否 → NovaDb 足够

## 12. 结论：差异化清单

| 差异化能力 | 说明 |
| --- | --- |
| **纯 .NET 托管** | 无 Native 依赖，支持 .NET Framework 4.5+，部署即 xcopy，与 .NET 应用同进程零序列化开销 |
| **嵌入+服务双模** | 开发调试嵌入如 SQLite，生产部署独立服务如 MySQL，同一套 API |
| **集群与主从同步** | 独立服务支持集群部署，支持一主多从同步，用于读扩展与容灾；多主多从作为远期目标 |
| **文件夹即数据库** | 拷贝文件夹完成迁移/备份，无需 dump/restore 流程 |
| **冷热分离索引** | 1000 万行表仅查热点时内存 < 20MB，冷数据自动卸载至 MMF |
| **关系型 + 时序 + MQ + KV + 向量 + 地理融合** | 单一引擎覆盖 SQLite + TDengine + Redis + Milvus + PostGIS 常见场景，减少组件数与集成复杂度 |
| **向量与关系数据无缝融合** | 同一 SQL 查询中支持向量 KNN 搜索与关系型过滤，避免应用层两次查询，提升开发效率 |
| **地理位置与业务数据统一** | 原生支持 GeoPoint 类型与地理函数（距离/范围/最近邻），无需 PostGIS，轻量级部署 |
| **增量物化视图 + OLAP** | 时序数据实时聚合 + 定时修正，支持多粒度汇总，无需外部流处理系统 |
| **强调可诊断/可验收** | EXPLAIN、慢查询、错误码稳定、系统表 SQL 查询 |
| **NewLife XCode 原生集成** | ORM 层直接适配，无需第三方驱动 |
| **明确非目标** | 不与全功能 SQL、专业 OLAP、分布式强一致正面竞争 |

补充说明：

- **同库联合查询思路**：倾向把“消息流水/时序明细/聚合结果”放在同一引擎内可查询。
- **后续可扩展模型**：向量/地理等作为可选能力讨论，但不会抢专业系统定位。

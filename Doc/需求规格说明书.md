# NewLife.NovaDb 需求规格说明书

## 1. 产品概述
*   **全称**：NewLife.NovaDb（诺娃数据库）
*   **简称**：Nova（诺娃）
*   **定位**：以 **C#** 实现的中大型混合数据库，支持嵌入式/服务器双模，融合关系型、时序、消息队列、NoSQL(KV) 能力。
*   **运行平台**：引擎与客户端驱动（ADO.NET Provider）均支持 **.NET Framework 4.5+**（含 .NET Core / .NET 5~10），纯托管代码，运行于 .NET 支持的所有操作系统（Windows/Linux/macOS）与指令集（x86/x64/ARM/ARM64）。
*   **愿景**：NewLife 生态核心基础设施，替代 SQLite/MySQL/Redis/TDengine 在特定场景的使用，一套引擎覆盖关系型、时序、MQ、KV 四大场景。
*   **约束**：纯托管代码（C#/.NET），不依赖 Native 组件。
*   **目标**：裁剪冷门能力，聚焦优化核心功能，通过简化架构降低复杂度，实现极致性能与更低运维成本；数据量逻辑上无上限（受磁盘与切分策略约束）。

### 1.1 核心设计理念

传统数据库面临"不可能三角"：**容量大**（支持 TB 级数据）、**性能快**（毫秒级查询）、**内存省**（不依赖海量内存）三者难以兼得。NovaDb 通过**分层存储 + 智能冷热分离**架构打破这一限制：

| 设计目标 | 实现策略 | 技术收益 |
|---------|---------|---------|
| **大容量** | 数据文件只占磁盘，按需加载页面到内存，单表支持 TB 级数据 | 突破物理内存限制，存储成本低 |
| **高性能** | 热点数据自动缓存，索引常驻内存，冷数据按需加载 | 热点查询毫秒级响应，媲美全内存数据库 |
| **低内存** | 冷数据主动逐出，内存占用可配置上限，避免 OOM | 单机可运行更多实例，云环境成本更低 |

**核心机制**：NovaDb 在存储层实现**冷热数据自动识别与分级管理**，频繁访问的热点数据保持在内存缓存中获得极致性能，长期未访问的冷数据主动逐出释放内存。这种分层架构让 NovaDb 既能像 SQLite 一样处理超大文件，又能像 Redis 一样快速响应热点查询，同时内存占用可控在百 MB 量级。

### 1.2 需求分级与表述约定

- **P0**：v1 必须具备，缺失即不可发布
- **P1**：v1 应具备，可在同一大版本内补齐
- **P2**：远期规划，不进入 v1 验收范围

表述用语：

- **必须**：强制要求（可验收）
- **应该**：默认要求（可验收，但允许在特定场景豁免）
- **可以**：可选能力（不作为 v1 必选验收项）

## 2. 核心架构

| 模块 | 规格说明 | 设计意图 |
| :--- | :--- | :--- |
| **部署模式** | **嵌入模式** (本地文件夹) / **服务器模式** (独立进程 TCP/IPv6，支持一主多从集群) | 嵌入如 SQLite，网络如 MySQL |
| **文件管理** | **文件夹即数据库**，每表独立文件组（数据/索引/WAL/锁文件等），无需分片 | 拷贝文件夹即完成迁移/备份 |
| **存储能力** | 支持 **超大表**,单表数据量可远超物理内存限制,数据文件只占磁盘,按需加载到内存(实现方式不限) | **又要大**：突破内存限制,支持 TB 级单表数据 |
| **索引策略** | 支持 **动态冷热分离** 索引能力：频繁访问的热点数据通过缓存加速,很少访问的冷数据主动逐出内存 | **又要快 + 省内存**：热点查询毫秒级响应,内存占用可控 |
| **并发模型** | 引擎内部统一 **MVCC 多读多写**（行级不阻塞），嵌入模式额外增加**表级文件锁**（跨进程单写协调）；两种模式共享同一套引擎代码 | 每表独立文件组让锁粒度缩小到单表，远优于 SQLite 库级锁 |
| **WAL 日志** | 支持 **FULL** (同步) / **NORMAL** (准实时异步) / **NONE** (异步) 三种持久性等级；表级独立 WAL | 兼顾可靠性与性能，支持按场景选择 |
| **Binlog** | 支持数据库级逻辑日志（可滚动/可清理）用于主从复制、增量备份、审计追踪 | 服务器模式可默认启用，嵌入模式可按需启用 |
| **只读模式** | 支持 **ReadOnly** 连接，禁止所有写操作 (DDL/DML) | 提升只读查询性能并规避多进程写冲突 |
| **通信接口** | 服务器模式通过 TCP/IPv6 对外提供访问接口，支持数据压缩能力 | 提升吞吐，减少网络传输 |
| **字符集** | 统一使用 **UTF-8** 编码，排序规则参考主流数据库实现（大小写不敏感等） | 简化实现，避免编码兼容问题 |

## 3. 引擎与存储目标

本章仅描述 NovaDb 需要达到的**能力与结果**，不描述具体实现方式。实现细节分别见：

- 存储实现细节：见[存储架构](存储架构.md)
- 引擎/SQL/MQ/KV 等实现细节：见[架构设计文档](架构设计文档.md)

### 3.1 Nova Engine（通用关系型）

- **容量**：单表行数不设逻辑上限，上限受磁盘与配置约束。
- **持久化**：支持可靠写入（可配置持久性等级），崩溃后可恢复到一致状态。
- **索引**：支持主键索引与二级索引；支持索引冷热分离能力以平衡内存与性能。
- **并发**：支持 MVCC 多读多写，读写尽量不互相阻塞。
- **适用**：通用 CRUD（订单、用户、配置等业务表）。

### 3.2 Flux Engine（时序 + MQ）

#### 3.2.1 时序能力
- **按时间组织**：支持按时间组织/管理数据（可分片），适合追加写入与时间范围查询。
- **降采样**：支持对历史数据进行降采样（Down-Sampling），降低存储与查询成本。
- **连续查询**：支持连续查询（Continuous Query），自动对新增数据执行聚合计算。
- **时间窗口聚合**：支持滚动窗口、滑动窗口等时间窗口聚合查询。
- **数据保留**：支持按时间或容量进行自动清理（TTL/保留策略）。

#### 3.2.2 MQ（Stream）能力
- **消费模型**：支持消费组、消费者、Pending、Ack 机制，参考 Redis Stream 设计。
- **消息顺序**：保证**严格顺序投递**，单表内消息按写入顺序消费（不支持分区）。
- **位点管理**：消费者位点**自动管理**，支持从指定位置开始消费。
- **阻塞读取**：支持阻塞读取与超时机制。
- **可靠投递**：支持延迟消息、死信队列（DLQ）、重试机制。
- **容量规模**：单表单日支持最大 **100 亿行**消息量（基于时序引擎按天分表实现）；超大规模需求由应用层拆分多表。

#### 3.2.3 适用场景
日志/指标/IoT 采集、内部任务队列、审计流水、事件溯源。

### 3.3 KV 模式（键值视图）

- **数据模型**：Key/Value + TTL（到期自动不可见/可清理），适用于缓存与会话等场景。
- **多表**：支持同一数据库内创建多个 KV 表，互相隔离，支持为不同表配置不同默认 TTL。
- **接口**：同时支持 SQL 与 API 方式访问（如 Get/Set/Add/Inc/Remove）。

### 3.4 存储与文件管理

- **文件夹即数据库**：拷贝目录即可完成迁移/备份。
- **每表独立文件组**：表间隔离，降低锁粒度。
- **WAL**：用于崩溃恢复与持久性控制；只读模式不产生写入相关的副作用。
- **Binlog（可选）**：用于主从复制、增量备份与审计追踪，支持滚动与清理策略。

## 4. 数据类型与函数

### 4.1 数据类型

| 类别 | 类型定义 | C# 映射 | 说明 |
| :--- | :--- | :--- | :--- |
| **布尔** | `BOOL` | `Boolean` | 真/假 |
| **整数** | `INT` | `Int32` | 32 位整数 |
| | `LONG` / `BIGINT` | `Int64` | 64 位整数 |
| **浮点** | `DOUBLE` | `Double` | 64 位双精度浮点数（IEEE 754），15-16 位十进制有效数字，二进制浮点表示 |
| **定点** | `DECIMAL` | `Decimal` | 128 位高精度定点数，29 位十进制有效数字，0-28 位小数，十进制精确存储，**统一精度**，无需声明 `(p,s)` |
| **字符串** | `STRING(n)` | `String` | UTF-8 编码，**必须指定长度** `n`（字节数），用于索引优化与存储分配 |
| | `STRING` | `String` | 无长度限制（动态），**不建议用于索引列** |
| **二进制** | `BINARY(n)` / `VARBINARY(n)` | `Byte[]` | **必须指定长度** `n`（字节数） |
| | `BLOB` | `Byte[]` | 大对象，无长度限制（动态） |
| **时间** | `DATETIME` | `DateTime` | 8 字节（Int64 Ticks），精确到 100 纳秒，范围 0001-01-01 至 9999-12-31，与 C# DateTime 完全一致 |
| **地理编码** | `GEOPOINT` | 自定义结构 | 经纬度坐标 `(Lat, Lon)` |
| **向量** | `VECTOR(n)` | `Single[]` | 定长浮点数组，用于 AI 向量检索 |

#### 4.1.1 类型设计原则
*   **精度统一**：`DECIMAL` 固定 128 位，开发者无需关心精度/标度，引擎内部保证精度足够业务使用。
*   **长度必需**：`STRING(n)` / `BINARY(n)` 必须指定长度，便于引擎优化索引（定长索引性能更高）与页布局。
*   **动态类型**：`STRING` / `BLOB` 无长度限制，但**不适合作为索引列**（索引需截断或哈希）。
*   **NULL 约束**：所有类型默认允许 `NULL`，支持 `NOT NULL` 约束（`CREATE TABLE` 时声明）。

#### 4.1.2 地理编码类型 (GeoPoint)
*   **目标**：支持地理坐标类型，并提供距离/范围判断等常用函数。
*   **查询能力**：
    - 距离查询：计算两点间距离（支持米/千米单位）
    - 范围查询：查询指定点半径范围内的所有点
    - 多边形查询：判断点是否在多边形区域内
    - 最近邻查询：查找离指定点最近的 N 个点
*   **性能要求**：支持地理索引，百万级数据下范围查询响应时间 < 100ms（P95）。
*   **适用场景**：LBS 应用、附近查询、地理围栏、物流轨迹。
*   **实现说明**：具体存储/索引/算法见[架构设计文档](架构设计文档.md)。

#### 4.1.3 向量类型 (Vector)
*   **目标**：支持定长向量类型与常用相似度/距离函数，并支持 KNN 查询能力。
*   **查询能力**：
    - 相似度算法：欧氏距离（L2）、余弦相似度、内积（Dot Product）
    - KNN 查询：支持 Top-K 最相似向量查询（K 值上限建议 1000）
    - 向量维度：支持 1~4096 维浮点向量（具体上限可配置）
*   **性能要求**：百万级向量下 KNN(K=10) 查询响应时间 < 500ms（P95）。
*   **适用场景**：语义搜索、推荐系统、RAG 向量检索、图像/音频特征匹配。
*   **实现说明**：具体存储/索引/算法见[架构设计文档](架构设计文档.md)。

### 4.2 SQL 函数支持（概要）

为保持需求文档的简洁性，关于具体函数的逐条清单及实现细节已移至独立文档：[SQL函数支持清单.md](SQL函数支持清单.md)。

## 5. SQL 与功能支持

本章以“对外可验收能力”为准，避免描述内部实现策略。

### 5.1 SQL 支持范围（P0）

- **P0-必须：DDL**
	- `CREATE DATABASE` / `DROP DATABASE`
	- `CREATE TABLE` / `DROP TABLE`
	- `ALTER TABLE`：至少支持 `ADD COLUMN` / `MODIFY COLUMN` / `DROP COLUMN` / `COMMENT`
	- `CREATE INDEX` / `DROP INDEX`（含 `UNIQUE INDEX`）
- **P0-必须：DML**
	- `INSERT` / `UPDATE` / `DELETE`
	- `UPSERT`（如语法存在差异，需在文档中明确方言）
	- `TRUNCATE`
- **P0-必须：查询**
	- `SELECT` + `WHERE` + `ORDER BY` + `GROUP BY` + `HAVING` + `LIMIT`/`OFFSET`
	- 参数化查询（ADO.NET 参数）

### 5.2 连接与子查询（P1）

- **P1-应该：JOIN**
	- 支持 `INNER JOIN`、`LEFT JOIN`、`RIGHT JOIN` 的基础语法
	- 支持多表连续 JOIN
- **P1-应该：子查询**
	- 支持 `IN` / `EXISTS` 的基础形态

### 5.3 查询优化与诊断（P0）

- **P0-必须：执行计划**
	- 支持 `EXPLAIN` 命令查看 SQL 执行计划
	- 显示索引使用情况、预估行数、执行顺序等关键信息
- **P0-必须：慢查询日志**
	- 支持慢查询阈值配置（如 > 1000ms）
	- 记录慢查询 SQL、执行时间、扫描行数等诊断信息
- **P1-应该：统计信息**
	- 支持查询表统计信息：行数、索引基数、数据文件大小
	- 支持通过系统表或 `SHOW` 命令访问

### 5.4 事务与一致性（P0）

- **P0-必须：事务语义**
	- 支持 `BEGIN`/`COMMIT`/`ROLLBACK`（或等价 API）
	- 隔离级别至少为 **Read Committed**，其对外语义必须明确：
		- 查询不得读取到其它事务的未提交数据
		- 已提交数据对后续语句可见（是否语句级/事务级一致性需明确，v1 至少保证 Read Committed）
	- 未显式开启事务时，语句应以自动提交方式执行（等价于每条语句一个独立事务）
- **P0-必须：崩溃恢复**
	- 发生异常退出后，重启必须恢复到一致状态：已提交事务可见，未提交事务不可见

### 5.5 只读模式（P0）

- **P0-必须：只读连接**
	- 只读连接必须拒绝所有写操作（DDL/DML），并返回明确错误
	- 只读连接允许查询与元数据读取（如 `SHOW`/`DESCRIBE`/系统表等，具体清单可在 SQL 文档中定义）

### 5.6 Binlog 管理（P0）

- **P0-必须：Binlog 管理命令**
	- `SHOW BINLOGS`
	- `SHOW BINLOG STATUS`
	- `PURGE BINLOGS TO/BEFORE`
	- `FLUSH BINLOGS`

### 5.7 Flux / MQ 扩展（P1）

- **P1-应该：消费组与可靠性**
	- At-Least-Once：读取后进入 Pending，业务成功后 Ack
	- 支持阻塞读取与超时
	- 支持延迟消息与死信队列（DLQ）
	- 位点自动管理（参考 Redis Stream）
	- 保证严格顺序投递（单表不分区）

### 5.8 KV 扩展（P1）

- **P1-应该：KV 表能力**
	- Key/Value + TTL 的对外语义：过期后必须对读取不可见，并支持清理
	- 支持 SQL 与 API 两种访问方式

### 5.9 明确非目标（v1）

- **P0-必须：不支持的 SQL 特性需明确并保持一致**
	- 不支持：视图、触发器、存储过程
	- 不支持：外键（FOREIGN KEY）、CHECK 约束（v1）
	- 窗口函数等复杂特性：v1 不承诺（如后续支持需单独列入 P1/P2）

### 5.10 ADO.NET Provider 与协议（P0）

- **P0-必须：ADO.NET 基础能力**
	- 提供 `DbProviderFactory` 入口，支持标准 ADO.NET 使用方式
	- 支持 `DbConnection`/`DbCommand`/`DbDataReader`/`DbParameter`/`DbTransaction` 的核心用法
	- 支持参数化查询与执行（防止拼接 SQL 的注入风险）
	- 支持命令超时与取消（如 API/语义存在差异需明确）
- **P0-必须：连接字符串**
	- 提供连接字符串构造器或等价机制
	- 支持指定嵌入模式路径与服务器模式地址/端口（具体字段名可在驱动文档中定义）
- **P0-必须：网络通信**
	- 服务器模式支持 TCP/IPv6 通信，并支持压缩能力以降低网络传输

### 5.11 运维与可观测性（P1）

- **P1-应该：监控指标**
	- **性能指标**：查询/插入/更新/删除操作的 QPS、响应时间（P50/P95/P99）
	- **IO 吞吐**：磁盘读写字节数/次数、WAL 写入速率、页面换入换出
	- **网络吞吐**（服务器模式）：接收/发送字节数、连接数、压缩率
	- **资源占用**：内存使用量、CPU 使用率、文件句柄数
	- **业务指标**：表行数、索引大小、慢查询计数、事务冲突/回滚次数、MQ 堆积量
- **P1-应该：系统表/视图**
	- 支持查询表列表、索引列表、连接列表、事务列表
	- 支持查询 WAL/Binlog 状态、数据文件统计
	- 支持通过 `SHOW` 命令或系统表访问元数据
- **P1-应该：备份与恢复**
	- 支持"文件夹即数据库"的离线备份/迁移（拷贝目录即可）
	- 支持基于 Binlog 的增量恢复能力（如纳入 v1，则升级为 P0）

### 5.12 安全（P2）

- **P2-可以：认证与鉴权**
	- 服务器模式可支持认证（账号/Token 等）与权限控制（库/表粒度）
	- 传输层加密（TLS）可作为扩展能力
- **明确不支持**：v1 **不支持**操作审计日志与静态数据加密（at-rest）

## 6. 性能与容量目标

### 6.1 性能基准（P0）

以下指标为**单节点嵌入模式**基准，服务器模式受网络影响会有所下降。

| 指标类型 | 目标值 | 测试条件 |
| :--- | :--- | :--- |
| **点查询（主键）** | P95 < 5ms | 百万行表，热数据在内存 |
| **范围查询** | P95 < 50ms | 返回 100 行，有索引 |
| **插入（单条）** | > 10,000 QPS | WAL=NORMAL，无索引 |
| **插入（批量）** | > 100,000 行/秒 | 批量插入 1000 行，WAL=NORMAL |
| **更新（主键）** | > 5,000 QPS | 单字段更新，WAL=NORMAL |
| **删除（主键）** | > 5,000 QPS | WAL=NORMAL |
| **JOIN 查询** | P95 < 200ms | 两表各 10 万行，有索引 |
| **聚合查询** | P95 < 500ms | 百万行表，`COUNT`/`SUM`/`AVG` |

### 6.2 容量限制（P0）

| 限制项 | 建议值 | 硬限制 | 说明 |
| :--- | :--- | :--- | :--- |
| **单表行数** | < 10 亿行 | 无逻辑上限 | 受磁盘与配置约束，超大表性能下降 |
| **单行大小** | < 1MB | 16MB | 超过建议值影响性能 |
| **单表索引数** | < 10 个 | 32 个 | 过多索引影响写入性能 |
| **索引键长度** | < 256 字节 | 4KB | 定长索引性能更优 |
| **数据库表数量** | < 1000 个 | 无限制 | 过多表影响元数据加载 |
| **单个数据库大小** | < 1TB | 无限制 | 取决于磁盘空间 |
| **并发连接数**（服务器） | < 1000 个 | 10000 个 | 受内存与 CPU 限制 |
| **并发事务数** | < 500 个 | 无限制 | 过多事务影响 MVCC 性能 |
| **向量维度** | < 2048 维 | 4096 维 | 高维向量影响查询性能 |
| **MQ 单表单日消息量** | < 100 亿行 | 无限制 | 基于 KV 引擎按天分表 |

### 6.3 资源占用预期（P1）

| 资源类型 | 基准值 | 说明 |
| :--- | :--- | :--- |
| **内存占用（空载）** | < 100MB | 嵌入模式最小占用 |
| **内存占用（工作集）** | 视索引大小而定 | 热索引全部加载，冷索引按需加载 |
| **CPU 使用率（空闲）** | < 1% | 无请求时接近 0 |
| **CPU 使用率（高负载）** | < 80% | 1000 QPS 持续写入 |
| **磁盘 IOPS** | 视 WAL 模式而定 | FULL 模式每次写入同步，NORMAL/NONE 异步批量 |

## 7. 版本兼容性（P0）

### 7.1 数据文件兼容性

- **P0-必须：向后兼容**
	- 新版本数据库**必须能打开**旧版本创建的数据文件
	- 首次写入时**自动升级**为新版本文件格式
	- 升级过程必须安全（保留原数据，升级失败可回滚）
- **P0-必须：版本标识**
	- 数据文件头包含版本号，读取时检查兼容性
	- 不兼容版本**必须明确拒绝并提示**用户升级路径
- **明确不支持**：不支持降级（新版本文件无法被旧版本打开）

### 7.2 通信协议兼容性（服务器模式）

- **P0-必须：版本协商**
	- 客户端与服务器建立连接时**必须交换协议版本**
	- 协议版本不一致时，服务器尽力支持**相近的低版本客户端**（如 v1.1 服务器支持 v1.0 客户端）
	- 协议版本差距过大时**必须拒绝连接并返回明确错误**
- **P1-应该：兼容范围**
	- 同一大版本内（如 v1.x）保持协议兼容
	- 跨大版本（如 v1.x vs v2.x）可以不兼容，需明确升级指引

## 8. 集群与高可用（P1）

> **优先级说明**：集群功能整体为 **P1**（v1 应具备），但优先级低于核心引擎与 SQL 功能。

### 8.1 集群架构

- **P1-应该：一主多从**
	- 支持 **1 主 + N 从** 架构（N 建议 ≤ 5）
	- 主节点负责所有写操作，从节点复制数据用于读扩展与容灾
	- 基于 Binlog 进行主从同步（异步复制）
- **P1-应该：读写分离**
	- **应用层负责**读写分离逻辑（连接字符串区分主从）
	- 数据库层面**不提供**自动读写分离与负载均衡

### 8.2 数据一致性

- **P1-应该：最终一致性**
	- 主从之间为**最终一致性**（异步复制，存在延迟）
	- 提供查询从节点复制延迟的能力（Lag 监控）
- **明确不支持**：不支持同步复制、强一致性读取（v1）

### 8.3 故障切换（P2）

- **P2-可以：手动切换**
	- 支持**手动提升**从节点为主节点
	- 提供一致性检查工具，确保切换安全
- **明确不支持**：v1 **不支持**自动故障检测与自动切换（需外部工具如 Keepalived）
- **RTO/RPO 目标**（手动切换）：
	- RTO（恢复时间）< 5 分钟
	- RPO（数据丢失）< 1 秒（取决于 Binlog 复制延迟）

### 8.4 扩展性

- **明确不支持**：v1 不支持分库分表、水平分片、多主多写
- **应用层方案**：大规模场景通过应用层分库分表实现水平扩展

## 9. 客户端生态（P0）

### 9.1 语言支持

- **P0-必须：C# ADO.NET**
	- 提供完整的 ADO.NET Provider 实现
	- 支持 .NET Framework 4.5+ / .NET Core / .NET 5~10
- **明确不支持**：v1 **不提供** Python/Java/Go/JavaScript 等其他语言驱动

### 9.2 ORM 集成

- **P0-必须：ADO.NET 兼容**
	- 兼容所有**基于 ADO.NET** 的 ORM 框架
	- 重点支持：NewLife.XCode、Entity Framework (EF6)、Entity Framework Core (EF Core)、Dapper
- **P0-必须：NewLife XCode 原生集成**
	- 提供 XCode 原生适配，支持实体生成、CRUD、分页等核心功能
	- 支持 XCode 特有功能（如缓存、分表等）

## 10. NovaDb 差异化定位

> 与主流关系型数据库、时序数据库、NoSQL、消息队列的详细特性对比见[竞品分析文档](竞品分析.md)。

| 差异化能力 | 说明 |
| :--- | :--- |
| **纯 .NET 托管** | 无 Native 依赖，支持 .NET Framework 4.5+，部署即 xcopy，与 .NET 应用同进程零序列化开销 |
| **嵌入+服务双模** | 开发调试嵌入如 SQLite，生产部署独立服务如 MySQL，同一套 API |
| **集群与主从同步** | 独立服务支持集群部署，支持一主多从同步，用于读扩展与容灾；多主多从作为远期目标 |
| **文件夹即数据库** | 拷贝文件夹完成迁移/备份，无需 dump/restore 流程 |
| **冷热分离索引** | 支持冷热分离索引能力，在热点查询场景显著降低内存占用 |
| **关系型 + 时序 + MQ + KV 融合** | 单一引擎覆盖 SQLite + TDengine + Redis 常见场景，减少组件数 |
| **NewLife XCode 原生集成** | ORM 层直接适配，无需第三方驱动 |

# NewLife.NovaDb 需求规格说明书

## 1. 产品概述
*   **全称**：NewLife.NovaDb
*   **简称**：Nova
*   **定位**：以 **C#** 实现，运行于 **.NET 平台**（支持 .NET Framework 4.5 ~ .NET 10）的中大型混合数据库，支持嵌入式/服务器双模，融合关系型、时序、消息队列、NoSQL(KV) 能力。
*   **愿景**：NewLife 生态核心基础设施，替代 SQLite/MySQL/Redis/TDengine 在特定场景的使用。
*   **约束**：纯托管代码（C#/.NET），不依赖 Native 组件。
*   **目标**：通过裁剪大量冷门能力（如存储过程/触发器/窗口函数等），换取更高的读写性能与更低的运维成本；数据量逻辑上无上限（受磁盘与切分策略约束）。

## 2. 核心架构

| 模块 | 规格说明 | 设计意图 |
| :--- | :--- | :--- |
| **部署模式** | **嵌入模式** (本地文件夹) / **服务器模式** (独立进程 TCP，支持集群与主从同步) | 嵌入如 SQLite，网络如 MySQL |
| **文件管理** | **文件夹即数据库**，每表独立文件组 (`.data`, `.idx`, `.wal`) | 拷贝文件夹即完成迁移/备份 |
| **存储介质** | **MMF (内存映射文件)** 为主，物理内存为辅 | 突破物理内存限制，支持超大表 |
| **索引策略** | **动态冷热分离** (见 3.1 节) | 热点快，冷点省内存 |
| **WAL 日志** | **FULL** (同步) / **NORMAL** (异步 1s) / **NONE** (异步) | 全局配置，默认 NORMAL |
| **只读模式** | 嵌入模式支持 **ReadOnly** 连接，禁止所有写操作 (DDL/DML) | 跳过 WAL 初始化、关闭脏页刷盘，提升读取性能；规避多进程并发写冲突；适用于报表查询、数据分析、从库只读副本等场景 |
| **通信协议** | 自定义二进制协议 (无压缩) | 极致性能，ADO.NET 驱动自动识别 |

## 3. 存储引擎详解

### 3.1 默认引擎：Nova Engine (通用关系型)
*   **数据结构**：SkipList (跳表)。
*   **容量目标**：单表支持 **无限行**（上限取决于磁盘空间与文件切分策略）。
*   **内存优化 (核心)**：
    *   **热数据 (Hot)**：最近访问 (如最近 10 分钟) 的数据段，索引完整加载至物理内存 (SkipList 节点)。
    *   **冷数据 (Cold)**：长期未访问数据，索引卸载至 MMF，内存仅保留**稀疏目录** (如每 1 万行一个锚点)。
    *   **效果**：1000 万行表，若只查最新 1 万行，内存占用 < 20MB；若全表随机查，依赖 MMF 页 fault，性能下降但可用。
    *   **淘汰机制**：LRU 策略，超过阈值 (如 30 分钟未访问) 自动卸载热索引。
*   **持久化与切分 (核心)**：
    *   **单表无上限**：逻辑上单表不设行数上限，数据增长主要消耗磁盘资源。
    *   **默认文件布局**：每表独立文件组（`.data`, `.idx`, `.wal`）。
    *   **自动文件切分**：当单个物理文件达到阈值（按大小/时间/行数等策略）后，自动切分为多个数据分片（Shard），以支持无上限存储。
    *   **冷热加载原则**：表数据极少读取或不读取时，不应占用显著内存；只有频繁读取的热点范围，才提升为内存热索引。
*   **索引文件 (核心)**：
    *   **多索引多文件**：一张表每个索引（主键/二级索引/唯一索引等）对应独立索引文件（主键索引为 `{TableName}.idx`，二级索引为 `{TableName}_{IndexName}.idx`），便于独立加载、淘汰与重建。
    *   **idx 不是内存跳表节点镜像**：`.idx` 存储可持久化的“目录 + 块/页”结构（稀疏目录锚点 -> 块偏移），用于按需 MMF 映射与快速定位；内存 SkipList 用作热缓存结构，可从 `.idx` 按需重建热点区域。
    *   **文件组命名**：每表文件组使用相同表名前缀，通过后缀区分（`.data`/`.wal`/`.idx`），分片时使用 `{TableName}_{ShardId}.{ext}` 格式，无需独立目录。
*   **适用场景**：通用 CRUD，配置表，业务订单，用户数据。

### 3.2 时序引擎：Flux Engine (时序 + MQ)
*   **数据结构**：按时间分片 (如每天/每小时一个文件)，追加写 (Append Only)。
*   **容量目标**：单表支持 **无限行** (依赖磁盘空间)。
*   **MQ 功能 (Stream)**：
    *   **模型**：参考 Redis Stream 的消费组模型（`Topic/Stream` + `ConsumerGroup` + `Consumer` + `Pending`）。
    *   **消息 ID**：时间戳 + 序列（同毫秒内自增），整体单调递增。
    *   **可靠性**：At-Least-Once，读取后进入 Pending，业务成功后 Ack。
    *   **阻塞读取**：支持阻塞 + 超时（类似 `XREADGROUP BLOCK`）。
    *   **延迟消息**：
        *   **实现机制**：消息发送时指定 `DelayTime`（延迟时长，秒）或 `DeliverAt`（投递时间），消息写入延迟队列（专用索引）。
        *   **投递策略**：后台扫描线程按时间顺序轮询延迟队列，到期消息自动转入正常队列供消费。
        *   **应用场景**：订单超时自动取消、定时通知、重试退避策略。
    *   **死信队列 (Dead Letter Queue, DLQ)**：
        *   **触发条件**：消息消费失败达到最大重试次数（可配置，默认 3 次）或消费超时后，自动进入死信队列。
        *   **存储结构**：每个普通队列对应一个独立死信队列（`{QueueName}_DLQ`），保留原始消息 + 失败原因 + 失败次数。
        *   **处理机制**：
            *   死信队列可独立订阅，用于人工介入或异常告警。
            *   支持死信消息重新投递回原队列（Retry）或永久删除。
        *   **应用场景**：消息处理异常排查、消息丢失防护、业务降级处理。
    *   **可观测性**：系统表记录队列、消费组、消费者、Pending、延迟消息、死信队列等状态，便于 SQL 查询使用情况。
*   **数据保留**：支持 TTL (按时间/文件大小自动删除旧分片)。
*   **适用场景**：物联网传感器数据，日志收集，内部消息队列，审计日志，定时任务调度。

### 3.3 KV 模式 (逻辑视图)
*   **实现**：
    *   **引擎复用**：不单独建引擎，复用 Nova Engine，通过 API 屏蔽 SQL 细节。
    *   **Schema**：特殊结构的 Nova 表 (`Key` string PK, `Value` byte[], `TTL` datetime)。
*   **TTL 机制**：
    *   **强制过期**：所有 key **必须**设置过期时间（`TTL`），防止无限期驻留内存。
    *   **自动清理**：后台线程定期扫描并删除过期记录（`NOW() > TTL`），保持内存与磁盘高效。
    *   **默认策略**：未显式指定时使用默认 TTL（如 24 小时），可全局配置。
*   **接口**：
    *   **SQL**：`SELECT Value FROM KvTable WHERE Key = '...' AND TTL > NOW()`
    *   **API**：
        *   `Get(key)`：获取值（自动忽略过期）。
        *   `Set(key, value, ttl)`：设置/更新值及过期时间。
        *   `Add(key, value, ttl)`：仅当 key 不存在时添加（分布式锁场景）。
        *   `Remove(key)`：删除 key。
        *   `Inc(key, delta, ttl)`：原子递增（计数器场景，key 不存在时初始化为 delta）。
*   **适用场景**：分布式锁，缓存，会话存储，计数器，配置中心。

## 4. 数据类型与函数

### 4.1 数据类型

| 类别 | 类型定义 | C# 映射 | 说明 |
| :--- | :--- | :--- | :--- |
| **布尔** | `BOOL` | `Boolean` | 真/假，1 字节 |
| **整数** | `INT` | `Int32` | 32 位整数，4 字节 |
| | `LONG` / `BIGINT` | `Int64` | 64 位整数，8 字节 |
| **浮点** | `DOUBLE` | `Double` | 双精度浮点，8 字节 |
| **定点** | `DECIMAL` | `Decimal` | 128 位定点数，**统一精度**，无需声明 `(p,s)` |
| **字符串** | `STRING(n)` | `String` | UTF-8 编码，**必须指定长度** `n`（字节数），用于索引优化与存储分配 |
| | `STRING` | `String` | 无长度限制（动态），**不建议用于索引列** |
| **二进制** | `BINARY(n)` / `VARBINARY(n)` | `Byte[]` | **必须指定长度** `n`（字节数） |
| | `BLOB` | `Byte[]` | 大对象，无长度限制（动态） |
| **时间** | `DATETIME` | `DateTime` | 精确到 Ticks（100 纳秒），支持时序排序 |
| **地理编码** | `GEOPOINT` | 自定义结构 | 经纬度坐标 `(Lat, Lon)`，16 字节（双 `Double`） |
| **向量** | `VECTOR(n)` | `Single[]` | 定长浮点数组，用于 AI 向量检索，如 `VECTOR(1536)` |

#### 4.1.1 类型设计原则
*   **精度统一**：`DECIMAL` 固定 128 位，开发者无需关心精度/标度，引擎内部保证精度足够业务使用。
*   **长度必需**：`STRING(n)` / `BINARY(n)` 必须指定长度，便于引擎优化索引（定长索引性能更高）与页布局。
*   **动态类型**：`STRING` / `BLOB` 无长度限制，但**不适合作为索引列**（索引需截断或哈希）。
*   **NULL 约束**：所有类型默认允许 `NULL`，支持 `NOT NULL` 约束（`CREATE TABLE` 时声明）。

#### 4.1.2 地理编码类型 (GeoPoint)
*   **存储**：`struct { double Lat; double Lon; }`，共 16 字节。
*   **计算函数**：
    *   `DISTANCE(point1, point2)`：计算两点距离（米），Haversine 公式。
    *   `WITHIN_RADIUS(point, center, radius)`：判断点是否在圆形范围内。
*   **索引**：基于 GeoHash 或 R-Tree 的空间索引，支持范围查询（BoundingBox）。
*   **适用场景**：LBS 应用，附近商家/用户查询，地理围栏。

#### 4.1.3 向量类型 (Vector)
*   **存储**：定长 `float[]` 数组，如 `VECTOR(1536)` 占用 1536 × 4 = 6144 字节。
*   **计算函数**：
    *   `COSINE_SIMILARITY(vec1, vec2)`：余弦相似度（-1 到 1）。
    *   `EUCLIDEAN_DISTANCE(vec1, vec2)`：欧氏距离。
    *   `DOT_PRODUCT(vec1, vec2)`：点积。
*   **索引**：基于 HNSW (层次可导航小世界图) 或 IVF (倒排文件) 的向量索引，支持 KNN (K 近邻) 查询。
*   **适用场景**：AI 向量检索（语义搜索、推荐系统、RAG 应用），支持 OpenAI Embeddings、文本向量化等。

## 7. 关键设计决策确认

1.  **索引冷热分离**：
    *   **机制**：内存维护“热段 SkipList" + “冷段稀疏目录”。
    *   **触发**：基于访问时间 (LRU)，后台线程定期卸载冷索引。
    *   **目标**：1000 万行表，仅查最新 1 万行时，内存占用 < 20MB。
2.  **超大表支持**：
*   **上限**：逻辑上无限制（受磁盘空间与文件切分策略约束）。
*   **代价**：冷数据查询性能随数据量与命中分片范围下降（依赖 MMF 页 fault），可能出现秒级响应。
    *   **建议**：XCode 层仍建议对超大数据进行逻辑分表 (按时间/Hash)。
3.  **MQ 可靠性**：
    *   **模型**：At-Least-Once (至少一次)。
    *   **流程**：读取消息 -> 进入 Pending -> 业务处理 -> Ack/确认（服务端更新消费组偏移量并移除 Pending）。
    *   **风险**：崩溃重试可能导致重复消费，需业务层幂等。
    *   **参考**：整体模型参考 Redis Stream 的消费组语义（服务端维护每个消费组的消费进度与 Pending）。

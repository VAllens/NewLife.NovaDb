# NewLife.NovaDb 需求规格说明书

## 1. 产品概述
*   **全称**：NewLife.NovaDb（诺娃数据库）
*   **简称**：Nova（诺娃）
*   **定位**：以 **C#** 实现的中大型混合数据库，支持嵌入式/服务器双模，融合关系型、时序、消息队列、NoSQL(KV) 能力。
*   **运行平台**：引擎与客户端驱动（ADO.NET Provider）均支持 **.NET Framework 4.5+**（含 .NET Core / .NET 5~10），纯托管代码，运行于 .NET 支持的所有操作系统（Windows/Linux/macOS）与指令集（x86/x64/ARM/ARM64）。
*   **愿景**：NewLife 生态核心基础设施，替代 SQLite/MySQL/Redis/TDengine 在特定场景的使用，一套引擎覆盖关系型、时序、MQ、KV 四大场景。
*   **约束**：纯托管代码（C#/.NET），不依赖 Native 组件。
*   **目标**：通过裁剪大量冷门能力（如存储过程/触发器/窗口函数等），换取更高的读写性能与更低的运维成本；数据量逻辑上无上限（受磁盘与切分策略约束）。

## 2. 核心架构

| 模块 | 规格说明 | 设计意图 |
| :--- | :--- | :--- |
| **部署模式** | **嵌入模式** (本地文件夹) / **服务器模式** (独立进程 TCP/IPv6，支持一主多从集群) | 嵌入如 SQLite，网络如 MySQL |
| **文件管理** | **文件夹即数据库**，每表独立文件组 (`.nvd`, `.nvi`, `.wal`)，无需分片 | 拷贝文件夹即完成迁移/备份 |
| **存储介质** | **MMF (内存映射文件)** 为主，物理内存为辅 | 64 位系统虚拟地址空间 128TB，突破物理内存限制，支持超大表 |
| **索引策略** | **动态冷热分离** (见 3.1 节) | 热点快，冷点省内存 |
| **并发模型** | 引擎内部统一 **MVCC 多读多写**（行级不阻塞），嵌入模式额外增加**表级文件锁**（跨进程单写协调）；两种模式共享同一套引擎代码 | 每表独立文件组让锁粒度缩小到单表，远优于 SQLite 库级锁 |
| **WAL 日志** | **FULL** (同步) / **NORMAL** (异步 1s) / **NONE** (全异步)，表级独立 WAL | 写入时产生 `.wal`，checkpoint 后删除；只读模式不创建 `.wal` |
| **Binlog** | 数据库级逻辑日志，记录已提交事务操作（INSERT/UPDATE/DELETE/DDL），按文件滚动（详见 3.4.5 节） | 主从复制、增量备份、审计追踪；嵌入模式默认不启用，服务器模式默认启用 |
| **只读模式** | 嵌入模式支持 **ReadOnly** 连接，禁止所有写操作 (DDL/DML) | 跳过 WAL 初始化、关闭脏页刷盘，提升读取性能；规避多进程并发写冲突；适用于报表查询、数据分析、从库只读副本等场景 |
| **通信协议** | 自定义二进制协议 (TCP/IPv6)，较大数据包自动启用压缩 | 极致性能，大包自动压缩减少网络传输，ADO.NET 驱动自动识别 |

## 3. 存储引擎详解

### 3.1 默认引擎：Nova Engine (通用关系型)
*   **数据结构**：SkipList (跳表)，实现简单、并发友好，配合 MMF 冷热分离弥补内存开销劣势（与 B+ Tree / LSM Tree 的选型对比分析见[架构设计文档](架构设计文档.md)）。
*   **容量目标**：单表支持 **无限行**（上限取决于磁盘空间，64 位系统下 MMF 虚拟地址空间 128TB，无需分片）。
*   **内存优化 (核心)**：
    *   **热数据 (Hot)**：最近访问 (如最近 10 分钟) 的数据段，索引完整加载至物理内存 (SkipList 节点)。
    *   **冷数据 (Cold)**：长期未访问数据，索引卸载至 MMF，内存仅保留**稀疏目录** (如每 1 万行一个锚点)。
    *   **效果**：1000 万行表，若只查最新 1 万行，内存占用 < 20MB；若全表随机查，依赖 MMF 页 fault，性能下降但可用。
    *   **淘汰机制**：LRU 策略，超过阈值 (如 30 分钟未访问) 自动卸载热索引。
*   **持久化 (核心)**：
    *   **单表单文件**：每张表一个 `.nvd` 数据文件，逻辑上不设行数上限，数据增长主要消耗磁盘资源。10G、100G 的数据文件在 MMF 下没有问题，OS 只映射实际访问的页，物理内存按需分配。
    *   **不分片**：Nova Engine 不做自动数据分片。如需逻辑分表（按时间/Hash），由 XCode ORM 层在应用层面完成。
    *   **默认文件布局**：每表独立文件组（`.nvd` + 多个 `.nvi` + 写入时的 `.wal`）。
    *   **冷热加载原则**：表数据极少读取或不读取时，不应占用显著内存；只有频繁读取的热点范围，才提升为内存热索引。
*   **索引文件 (核心)**：
    *   **多索引多文件**：一张表每个索引（主键/二级索引/唯一索引等）对应独立索引文件（主键索引为 `{TableName}.nvi`，二级索引为 `{TableName}_{IndexName}.nvi`），便于独立加载、淘汰与重建。
    *   **nvi 不是内存跳表节点镜像**：`.nvi` 存储可持久化的“目录 + 块/页”结构（稀疏目录锚点 -> 块偏移），用于按需 MMF 映射与快速定位；内存 SkipList 用作热缓存结构，可从 `.nvi` 按需重建热点区域。
    *   **文件组命名**：每表文件组使用相同表名前缀，通过后缀区分（`.nvd`/`.wal`/`.nvi`），所有文件平铺在数据库目录下。
*   **并发控制**：引擎内部统一 MVCC 多读多写（行级不阻塞），嵌入模式额外增加表级文件锁（跨进程单写协调），远优于 SQLite 库级锁。详见 3.4.6 节。
*   **适用场景**：通用 CRUD，配置表，业务订单，用户数据。

### 3.2 时序引擎：Flux Engine (时序 + MQ)
*   **数据结构**：按时间分片 (如每天/每小时一个 `.nvd` 文件)，追加写 (Append Only)。
*   **容量目标**：单表支持 **无限行** (依赖磁盘空间)。时间分片自然限制单个文件大小。
*   **统一数据模型**：时序数据与 MQ 消息采用同一存储结构。**每一条 MQ 消息就是时序表的一行数据**（MessageId + Fields），Stream/Topic 名即时序表名。发送消息 = 向时序表追加一行，消费消息 = 按 MessageId 顺序读取行。
*   **MQ 功能 (Stream)**：
    *   **模型**：参考 Redis Stream 的消费组模型（`Topic/Stream` + `ConsumerGroup` + `Consumer` + `Pending`）。
    *   **消息 ID**：时间戳 + 序列（同毫秒内自增），整体单调递增。消息 ID 同时也是时序表的主键。
    *   **可靠性**：At-Least-Once，读取后进入 Pending，业务成功后 Ack。
    *   **阻塞读取**：支持阻塞 + 超时（类似 `XREADGROUP BLOCK`）。
    *   **延迟消息**：
        *   **实现机制**：消息发送时指定 `DeliverAt`（投递时间），消息写入延迟队列（专用索引）。
        *   **投递策略**：后台扫描线程按时间顺序轮询延迟队列，到期消息自动转入正常队列供消费。
        *   **应用场景**：订单超时自动取消、定时通知、重试退避策略。
    *   **死信队列 (Dead Letter Queue, DLQ)**：
        *   **触发条件**：消息消费失败达到最大重试次数（可配置，默认 3 次）或消费超时后，自动进入死信队列。
        *   **存储结构**：每个普通队列对应一个独立死信队列（`{QueueName}_DLQ`），保留原始消息 + 失败原因 + 失败次数。
        *   **处理机制**：
            *   死信队列可独立订阅，用于人工介入或异常告警。
            *   支持死信消息重新投递回原队列（Retry）或永久删除。
        *   **应用场景**：消息处理异常排查、消息丢失防护、业务降级处理。
    *   **可观测性**：系统表记录队列、消费组、消费者、Pending、延迟消息、死信队列等状态，便于 SQL 查询使用情况。
*   **数据保留**：支持 TTL (按时间/文件大小自动删除旧分片)。
*   **适用场景**：物联网传感器数据，日志收集，内部消息队列，审计日志，定时任务调度。

### 3.3 KV 模式 (逻辑视图)
*   **实现**：
    *   **引擎复用**：不单独建引擎，复用 Nova Engine，通过 API 屏蔽 SQL 细节。
    *   **Schema**：固定结构的 Nova 表 (`Key` STRING(200) PK, `Value` BLOB, `TTL` DATETIME)。
    *   **多表支持**：一个数据库可以创建多个 KV 表，通过 `ENGINE=KV` 区分。不同 KV 表互相独立，可用于不同业务场景（如 `SessionCache`、`ConfigCache`、`DistLock`），每个表可配置不同的默认 TTL。
*   **创建方式**：
    *   **SQL**：`CREATE TABLE SessionCache (Key STRING(200) PRIMARY KEY, Value BLOB, TTL DATETIME) ENGINE=KV`
    *   **API**：`db.CreateKvStore("SessionCache", defaultTtl: TimeSpan.FromHours(2))`
*   **TTL 机制**：
    *   **强制过期**：所有 key **必须**设置过期时间（`TTL`，绝对 UTC 时间表示到期），防止无限期驻留内存。
    *   **自动清理**：后台线程定期扫描并删除过期记录（`NOW() > TTL`），保持内存与磁盘高效。
    *   **默认策略**：未显式指定时使用 KV 表级默认 TTL（如 24 小时），可在建表时配置。
*   **接口**：
    *   **SQL**：`SELECT Value FROM SessionCache WHERE Key = '...' AND TTL > NOW()`
    *   **API**：
        *   `Get(key)`：获取值（自动忽略过期）。
        *   `Set(key, value, ttl)`：设置/更新值及过期时间。
        *   `Add(key, value, ttl)`：仅当 key 不存在时添加（分布式锁场景）。
        *   `Remove(key)`：删除 key。
        *   `Inc(key, delta, ttl)`：原子递增（计数器场景，key 不存在时初始化为 delta）。
*   **适用场景**：分布式锁，缓存，会话存储，计数器，配置中心。

### 3.4 存储系统设计

#### 3.4.1 文件夹即数据库
每个数据库对应一个目录，目录下平铺所有文件。拷贝目录即完成数据库迁移/备份。

#### 3.4.2 数据库元数据文件 (`nova.db`)
二进制格式，程序可读即可，不需要人工读取。包含整个数据库的完整元数据（详细文件格式见存储架构文档）：
*   **数据库信息**：数据库名称、创建时间、最后修改时间。
*   **表目录**：所有表的定义，每张表包含：
    *   表名、引擎类型（Nova/Flux/KV）、注释、创建时间。
    *   列定义列表：列名、数据类型、长度、是否可空、是否主键、是否自增、默认值、注释。
    *   索引定义列表：索引名、是否唯一、包含的列名列表。
*   **修改机制**：CREATE/ALTER/DROP TABLE 时原子重写 `nova.db`（先写临时文件再原子重命名），保证崩溃安全。
*   **启动加载**：引擎启动时一次性读取 `nova.db`，在内存中构建表 Schema 缓存。

#### 3.4.3 数据库文件组织

所有文件平铺在数据库目录下，分为**数据库级文件**和**表级文件**两类。

**数据库级文件**：

| 文件 | 命名规则 | 说明 |
| :--- | :--- | :--- |
| 元数据文件 | `nova.db` | 数据库完整元数据（表/列/索引定义），二进制格式 |
| Binlog 日志 | `nova.{SeqNo:D6}.binlog` | 主从同步/增量备份的逻辑日志，数据库级（详见 3.4.5 节） |

**表级文件组**（每张表由以下文件组成）：

| 文件 | 后缀 | 说明 |
| :--- | :--- | :--- |
| 数据文件 | `.nvd` | 表的行数据，每表一个，通过 MMF 访问 |
| 主键索引 | `.nvi` | 主键索引文件，每表一个 |
| 二级索引 | `_{IndexName}.nvi` | 每个二级索引独立一个文件 |
| WAL 日志 | `.wal` | 写入时产生，checkpoint 后删除 |
| 写锁文件 | `.lock` | 嵌入模式下的表级写锁（服务器模式不需要） |

表级文件命名规则：
*   数据文件：`{TableName}.nvd`
*   主键索引：`{TableName}.nvi`
*   二级索引：`{TableName}_{IndexName}.nvi`
*   WAL 文件：`{TableName}.wal`
*   写锁文件：`{TableName}.lock`
*   时序分片（仅 Flux Engine）：`{TableName}_{yyyyMMdd}.nvd` 或 `{TableName}_{yyyyMMddHH}.nvd`

#### 3.4.4 WAL 生命周期

| 阶段 | 行为 |
| :--- | :--- |
| **写入** | 事务写入时，WAL 记录追加到 `{TableName}.wal` |
| **Checkpoint** | 后台线程将脏页刷新到 `.nvd`，完成后删除 `.wal` |
| **崩溃恢复** | 启动时发现 `.wal` 存在 → 重放已提交事务 → 恢复后删除 |
| **只读模式** | 不创建 `.wal` 文件，跳过 WAL 初始化 |

详细实现见[存储架构文档](存储架构.md)。

#### 3.4.5 Binlog 日志（主从同步/增量备份）

Binlog 是数据库级别的逻辑日志，记录所有已提交事务的逻辑操作（INSERT/UPDATE/DELETE/DDL），用于主从复制、增量备份和审计追踪。

**WAL 与 Binlog 的区别**：

| 维度 | WAL | Binlog |
| :--- | :--- | :--- |
| **层级** | 表级（每表一个 `.wal`） | 数据库级（全库共享 `.binlog`） |
| **记录内容** | 物理页面变更（UpdatePage） | 逻辑操作（INSERT/UPDATE/DELETE/DDL） |
| **用途** | 崩溃恢复 | 主从复制 + 增量备份 + 审计 |
| **生命周期** | 短暂（checkpoint 后删除） | 持久保留（按策略滚动清理） |
| **写入时机** | 事务写入时（提交前） | 事务提交后 |
| **可读性** | 二进制页面数据 | 结构化逻辑记录（可解析重放） |

##### 3.4.5.1 使用场景

| 场景 | 说明 |
| :--- | :--- |
| **主从复制** | 主节点写入 Binlog，从节点拉取并重放，实现数据同步 |
| **增量备份** | 备份工具记录上次备份的 LSN，下次仅拉取后续 Binlog 重放 |
| **审计追踪** | Binlog 记录了谁在什么时间对哪张表做了什么操作 |
| **数据恢复** | 全量备份 + Binlog 重放 = 恢复到任意时间点（Point-in-Time Recovery） |

##### 3.4.5.2 启用方式

| 模式 | 默认状态 | 配置方式 |
| :--- | :--- | :--- |
| **嵌入模式** | 不启用 | `DbOptions.EnableBinlog = true`，或连接字符串 `EnableBinlog=true` |
| **服务器模式** | 启用 | 默认启用，可通过配置关闭 |

##### 3.4.5.3 文件管理与清理

**文件命名**：`nova.{SeqNo:D6}.binlog`（如 `nova.000001.binlog`），序号永远递增不回绕，单文件达到大小阈值（默认 256MB）时自动滚动到新文件。

**清理策略**（可叠加，任一满足即清理）：

| 策略 | 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| **从节点确认** | 自动 | - | 所有从节点已确认同步的文件可安全清理 |
| **按天数保留** | `BinlogRetentionDays` | 0（不限） | 超过保留天数的旧文件自动删除 |
| **按总大小限制** | `BinlogMaxTotalSize` | 0（不限） | 总大小超过上限时从最早文件开始删除 |

每个 `.binlog` 文件自包含无交叉依赖，可安全手工删除；从节点支持断点续传（携带已同步 LSN 向主节点请求后续 Binlog，无需全量复制）。详细运维操作与主从复制流程见[存储架构文档](存储架构.md)。

##### 3.4.5.4 配置选项

| 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `EnableBinlog` | false (嵌入) / true (服务器) | 是否启用 Binlog |
| `BinlogMaxFileSize` | 256MB | 单文件滚动阈值 |
| `BinlogRetentionDays` | 0 | 保留天数，0 表示不按时间清理 |
| `BinlogMaxTotalSize` | 0 | 总大小上限（字节），0 表示不限制 |

#### 3.4.6 并发与锁

| 模式 | 引擎并发 | 跨进程锁 | 说明 |
| :--- | :--- | :--- | :--- |
| **嵌入模式** | MVCC 多读多写（进程内） | 表级文件锁 (`{TableName}.lock`)，跨进程单写多读 | 进程内 MVCC 提供行级不阻塞；跨进程通过文件锁协调写入，不同表互不影响 |
| **服务器模式** | MVCC 多读多写 | 无（单进程） | 所有请求通过 SqlEngine 统一调度，MVCC 保证行级读写不阻塞 |
| **只读模式** | 只读 | 无锁 | 不创建 `.wal` 和 `.lock`，多进程可同时以只读方式打开 |

**代码统一**：嵌入模式与服务器模式的引擎代码完全相同，均使用 MVCC 实现多读多写。表级文件锁仅是嵌入模式跨进程的附加协调层。

**与 SQLite 对比**：SQLite 使用库级锁，写入时锁定整个数据库文件，所有表读写都受影响。NovaDb 每表独立文件组让锁粒度缩小到单表，写 `User` 表不影响 `App` 表的读写，并发度显著提升。

## 4. 数据类型与函数

### 4.1 数据类型

| 类别 | 类型定义 | C# 映射 | 说明 |
| :--- | :--- | :--- | :--- |
| **布尔** | `BOOL` | `Boolean` | 真/假，1 字节 |
| **整数** | `INT` | `Int32` | 32 位整数，4 字节 |
| | `LONG` / `BIGINT` | `Int64` | 64 位整数，8 字节 |
| **浮点** | `DOUBLE` | `Double` | 双精度浮点，8 字节 |
| **定点** | `DECIMAL` | `Decimal` | 128 位定点数，**统一精度**，无需声明 `(p,s)` |
| **字符串** | `STRING(n)` | `String` | UTF-8 编码，**必须指定长度** `n`（字节数），用于索引优化与存储分配 |
| | `STRING` | `String` | 无长度限制（动态），**不建议用于索引列** |
| **二进制** | `BINARY(n)` / `VARBINARY(n)` | `Byte[]` | **必须指定长度** `n`（字节数） |
| | `BLOB` | `Byte[]` | 大对象，无长度限制（动态） |
| **时间** | `DATETIME` | `DateTime` | 精确到 Ticks（100 纳秒），支持时序排序 |
| **地理编码** | `GEOPOINT` | 自定义结构 | 经纬度坐标 `(Lat, Lon)`，16 字节（双 `Double`） |
| **向量** | `VECTOR(n)` | `Single[]` | 定长浮点数组，用于 AI 向量检索，如 `VECTOR(1536)` |

#### 4.1.1 类型设计原则
*   **精度统一**：`DECIMAL` 固定 128 位，开发者无需关心精度/标度，引擎内部保证精度足够业务使用。
*   **长度必需**：`STRING(n)` / `BINARY(n)` 必须指定长度，便于引擎优化索引（定长索引性能更高）与页布局。
*   **动态类型**：`STRING` / `BLOB` 无长度限制，但**不适合作为索引列**（索引需截断或哈希）。
*   **NULL 约束**：所有类型默认允许 `NULL`，支持 `NOT NULL` 约束（`CREATE TABLE` 时声明）。

#### 4.1.2 地理编码类型 (GeoPoint)
*   **存储**：`struct { double Lat; double Lon; }`，共 16 字节。
*   **计算函数**：
    *   `DISTANCE(point1, point2)`：计算两点距离（米），Haversine 公式。
    *   `WITHIN_RADIUS(point, center, radius)`：判断点是否在圆形范围内。
*   **索引**：基于 GeoHash 或 R-Tree 的空间索引，支持范围查询（BoundingBox）。
*   **适用场景**：LBS 应用，附近商家/用户查询，地理围栏。

#### 4.1.3 向量类型 (Vector)
*   **存储**：定长 `float[]` 数组，如 `VECTOR(1536)` 占用 1536 × 4 = 6144 字节。
*   **计算函数**：
    *   `COSINE_SIMILARITY(vec1, vec2)`：余弦相似度（-1 到 1）。
    *   `EUCLIDEAN_DISTANCE(vec1, vec2)`：欧氏距离。
    *   `DOT_PRODUCT(vec1, vec2)`：点积。
*   **索引**：基于 HNSW (层次可导航小世界图) 或 IVF (倒排文件) 的向量索引，支持 KNN (K 近邻) 查询。
*   **适用场景**：AI 向量检索（语义搜索、推荐系统、RAG 应用），支持 OpenAI Embeddings、文本向量化等。

### 4.2 SQL 函数支持（概要）

为保持需求文档的简洁性，关于具体函数的逐条清单及实现细节已移至独立文档：[Doc/SQL函数支持清单.md](Doc/SQL函数支持清单.md)。

本需求仅声明需支持的函数类别与兼容性原则：

- **必须覆盖的类别（示例）**：字符串、数值、日期/时间、聚合、条件表达式、类型转换、系统/元数据函数。
- **可选/扩展类别**：窗口函数、JSON 操作、地理（Geo）/向量（Vector）扩展、加密/哈希、正则与全文检索（按需评估）。
- **兼容性原则**：优先兼容 SQL-92 标准常用函数；对常见数据库（PostgreSQL/MySQL/SQL Server）重要扩展予以支持并在文档中标注来源与差异。

具体函数名称、语法、优先级（P0/P1/P2）、行为差异与测试用例参见独立清单。

### 4.3 约束支持

| 约束类型 | 支持度 | 说明 |
| :--- | :--- | :--- |
| **主键 (PRIMARY KEY)** | ✅ | 唯一且非 NULL，自动创建索引 |

## 6. NovaDb 差异化定位

> 与主流关系型数据库、时序数据库、NoSQL、消息队列的详细特性对比见[竞品分析文档](竞品分析.md)。

| 差异化能力 | 说明 |
| :--- | :--- |
| **纯 .NET 托管** | 无 Native 依赖，支持 .NET Framework 4.5+，部署即 xcopy，与 .NET 应用同进程零序列化开销 |
| **嵌入+服务双模** | 开发调试嵌入如 SQLite，生产部署独立服务如 MySQL，同一套 API |
| **集群与主从同步** | 独立服务支持集群部署，支持一主多从同步，用于读扩展与容灾；多主多从作为远期目标 |
| **文件夹即数据库** | 拷贝文件夹完成迁移/备份，无需 dump/restore 流程 |
| **冷热分离索引** | 1000 万行表仅查热点时内存 < 20MB，冷数据自动卸载至 MMF |
| **关系型 + 时序 + MQ + KV 融合** | 单一引擎覆盖 SQLite + TDengine + Redis 常见场景，减少组件数 |
| **NewLife XCode 原生集成** | ORM 层直接适配，无需第三方驱动 |

# NewLife.NovaDb 需求规格说明书

## 1. 产品概述

- **全称**：NewLife.NovaDb（诺娃数据库）
- **简称**：Nova（诺娃）
- **定位**：纯 **C#** 实现的混合数据库，支持嵌入式/网络双模，融合**关系型（Nova）**、**时序+MQ（Flux）**、**KV** 三大引擎。
- **平台**：.NET Framework 4.5+（含 .NET Core / .NET 5~10），纯托管代码，运行于 .NET 支持的所有操作系统与指令集。
- **愿景**：NewLife 生态核心基础设施，一套引擎覆盖关系型、时序、MQ、KV 四大场景，替代 SQLite/MySQL/Redis/TDengine 在特定场景的使用。

### 1.1 核心设计理念

传统数据库面临"不可能三角"：**容量大**、**性能快**、**内存省**三者难以兼得。NovaDb 通过**分层存储 + 智能冷热分离**打破这一限制：

| 设计目标 | 实现策略 | 技术收益 |
|---------|---------|---------|
| **大容量** | 数据文件只占磁盘，按需加载页面到内存，单表支持 TB 级 | 突破物理内存限制，存储成本低 |
| **高性能** | 热点数据自动缓存，索引常驻内存，冷数据按需加载 | 热点查询毫秒级响应 |
| **低内存** | 冷数据主动逐出，内存占用可配置上限 | 单机可运行更多实例 |

### 1.2 优先级定义

- **P0**：v1 必须具备，缺失即不可发布
- **P1**：v1 应具备，可在同一大版本内补齐
- **P2**：远期规划，不进入 v1 验收

## 2. 核心架构

| 模块 | 规格说明 | 设计意图 |
|------|---------|---------|
| **部署模式** | 嵌入模式（本地文件夹）/ 网络模式（独立进程 TCP/IPv6，支持一主多从） | 嵌入如 SQLite，网络如 MySQL |
| **文件管理** | **文件夹即数据库**，每表独立文件组，无需分片 | 拷贝文件夹即完成迁移/备份 |
| **存储能力** | 单表数据量可远超物理内存，数据文件只占磁盘，按需加载 | 突破内存限制，支持 TB 级 |
| **索引策略** | 冷热分离：热点数据缓存加速，冷数据主动逐出 | 热点查询毫秒级，内存可控 |
| **并发模型** | MVCC 多读多写（行级不阻塞），嵌入模式额外表级文件锁（跨进程单写） | 远优于 SQLite 库级锁 |
| **WAL 日志** | FULL（同步）/ NORMAL（异步）/ NONE（关闭）三种持久性 | 按场景选择可靠性与性能 |
| **Binlog** | 数据库级逻辑日志，可滚动/清理，用于主从复制、增量备份、审计 | 网络模式默认启用 |
| **只读模式** | 禁止所有写操作，不产生写副作用 | 提升查询性能，规避写冲突 |
| **通信** | TCP/IPv6，支持数据压缩 | 提升吞吐，减少传输 |
| **字符集** | 统一 UTF-8 | 简化实现 |

## 3. 引擎与存储

### 3.1 Nova Engine（关系型）

- **容量**：单表行数无逻辑上限，受磁盘与配置约束
- **持久化**：可配置持久性等级，崩溃后恢复到一致状态
- **索引**：主键索引 + 二级索引，支持冷热分离
- **并发**：MVCC 多读多写，读写不互相阻塞
- **适用**：通用 CRUD（订单、用户、配置等业务表）

### 3.2 Flux Engine（时序 + MQ）

#### 时序能力
- 按时间组织数据（可分片），适合追加写入与时间范围查询
- 降采样（Down-Sampling）降低存储与查询成本
- 连续查询（Continuous Query）自动对新增数据执行聚合
- 滚动窗口、滑动窗口等时间窗口聚合
- 按时间或容量自动清理（TTL/保留策略）

#### MQ（Stream）能力
- 消费组、消费者、Pending、Ack，参考 Redis Stream
- 严格顺序投递，单表内消息按写入顺序消费
- 消费者位点自动管理，支持从指定位置开始消费
- 阻塞读取与超时
- 延迟消息、死信队列（DLQ）、重试机制
- 单表单日最大 **100 亿行**（基于时序引擎按天分表）

#### OLAP 与物化视图
- 增量物化视图：持续聚合，自动增量更新统计表
- 定时修正：定时对源表 GROUP BY，修正延迟/乱序数据
- 实时聚合引擎：MQ 消费时自动聚合，批量更新统计表
- 多粒度汇总：1分钟 → 15分钟 → 1小时 → 1天
- 刷新策略：可配置缓冲区大小、刷新间隔、修正调度

#### 适用场景
日志/指标/IoT 采集、内部任务队列、审计流水、事件溯源、实时统计分析。

### 3.3 KV 模式（键值）

- **数据模型**：Key/Value + TTL，到期自动不可见/清理
- **多表**：同一数据库内多个 KV 表，互相隔离，不同默认 TTL
- **接口**：SQL + API（Get/Set/Add/Inc/Remove）

### 3.4 存储与文件管理

- **文件夹即数据库**：拷贝目录即迁移/备份
- **每表独立文件组**：表间隔离，降低锁粒度
- **WAL**：崩溃恢复与持久性控制
- **Binlog（可选）**：主从复制、增量备份与审计追踪

## 4. 数据类型

### 4.1 类型定义

| 类别 | 类型 | C# 映射 | 说明 |
|------|------|--------|------|
| 布尔 | `BOOL` | `Boolean` | 真/假 |
| 整数 | `INT` | `Int32` | 32 位 |
| | `LONG` / `BIGINT` | `Int64` | 64 位 |
| 浮点 | `DOUBLE` | `Double` | 64 位双精度 |
| 定点 | `DECIMAL` | `Decimal` | 128 位高精度，无需声明 (p,s) |
| 字符串 | `STRING(n)` | `String` | 必须指定长度（字节数），用于索引优化 |
| | `STRING` | `String` | 无长度限制，不建议用于索引列 |
| 二进制 | `BINARY(n)` / `VARBINARY(n)` | `Byte[]` | 必须指定长度 |
| | `BLOB` | `Byte[]` | 大对象，无长度限制 |
| 时间 | `DATETIME` | `DateTime` | 8 字节 Int64 Ticks，精确到 100ns |
| 地理 | `GEOPOINT` | 自定义结构 | 经纬度 (Lat, Lon)，16 字节 |
| 向量 | `VECTOR(n)` | `Single[]` | 定长浮点数组，AI 向量检索 |

#### 类型设计原则
- `DECIMAL` 固定 128 位，无需关心精度/标度
- `STRING(n)` / `BINARY(n)` 必须指定长度，便于索引优化
- 所有类型默认允许 `NULL`，支持 `NOT NULL` 约束
- `DATETIME` 不做时区换算，原样存储与返回

#### GeoPoint（P2）
- 距离查询、范围查询、多边形查询、最近邻查询
- 百万级范围查询 P95 < 100ms

#### Vector（P2）
- 欧氏距离（L2）、余弦相似度、内积
- KNN Top-K 查询，百万级 KNN(K=10) P95 < 500ms

### 4.2 SQL 函数

SQL 函数的逐条清单见 [SQL函数支持清单.md](SQL函数支持清单.md)。需覆盖的类别：

- **P0 必须**：字符串、数值、日期时间、聚合、条件表达式、类型转换、系统元数据
- **P1 应该**：哈希、地理扩展、向量扩展
- **P2 远期**：窗口函数、正则、全文检索

兼容性原则：优先兼容 SQL-92 标准；对 MySQL/PostgreSQL/SQL Server 重要扩展予以支持。

### 4.3 约束支持

| 约束类型 | 支持度 | 说明 |
|---------|--------|------|
| PRIMARY KEY | ✅ | 唯一且非 NULL，自动创建索引 |
| INDEX | ✅ | 普通索引、唯一索引 |
| UNIQUE INDEX | ✅ | 单字段/多字段联合唯一索引 |
| NOT NULL | ✅ | 列级约束 |
| DEFAULT | ✅ | 如 `DEFAULT 0`、`DEFAULT NOW()` |
| AUTO_INCREMENT | ✅ | 主键自增，仅 INT/LONG |
| FOREIGN KEY | ❌ | 不支持，由应用层保证 |
| CHECK | ❌ | v1 不实现 |

## 5. SQL 与功能支持

### 5.1 SQL 支持范围（P0）

**方言约定**：
- 关键字/标识符不区分大小写
- 双引号引用标识符（SQL-92），可选支持反引号
- 字符串用单引号，转义 `''`
- 命名参数 `@name`
- `--` 单行注释
- `ENGINE=NOVA|FLUX|KV` 选择引擎

**P0-DDL**：`CREATE/DROP DATABASE`、`CREATE/DROP TABLE`、`ALTER TABLE`（ADD/MODIFY/DROP COLUMN、COMMENT）、`CREATE/DROP INDEX`（含 UNIQUE）

**P0-DML**：`INSERT`、`UPDATE`、`DELETE`、`UPSERT`、`TRUNCATE`

**P0-查询**：`SELECT` + `WHERE` + `ORDER BY` + `GROUP BY` + `HAVING` + `LIMIT/OFFSET`、参数化查询

### 5.2 连接与子查询（P1）

- `INNER JOIN`、`LEFT JOIN`、`RIGHT JOIN`，多表连续 JOIN
- `IN` / `EXISTS` 基础子查询

### 5.3 事务与一致性（P0）

- 支持 `BEGIN`/`COMMIT`/`ROLLBACK`
- 隔离级别：**Read Committed**
  - 不读脏数据
  - 语句级一致性（允许 Phantom Read）
  - 读己之写
- 写写冲突返回明确可重试错误
- 未显式开启事务时自动提交
- DDL 为隐式提交（不可回滚）
- 崩溃恢复：已提交可见，未提交不可见

### 5.4 只读模式（P0）

只读连接拒绝所有 DDL/DML，允许查询与元数据读取。

### 5.5 Binlog 管理（P0）

`SHOW BINLOGS`、`SHOW BINLOG STATUS`、`PURGE BINLOGS TO/BEFORE`、`FLUSH BINLOGS`。清理幂等，越界返回明确错误。

### 5.6 Flux / MQ 扩展（P1）

- At-Least-Once：读取 → Pending → Ack
- 阻塞读取与超时
- 延迟消息与死信队列
- 位点自动管理，严格顺序投递
- **不提供** Exactly-Once（由业务侧幂等实现）

### 5.7 KV 扩展（P1）

- Key/Value + TTL，过期后读取不可见，惰性删除 + 主动清理
- 多表隔离，每个 KV 表独立 `.kvd` 文件，支持独立默认 TTL
- 原子操作：Add（仅不存在时写入）、Replace（替换并返回旧值）、Inc/IncDouble（原子计数器）
- 批量操作：GetAll、SetAll、批量 Delete
- 搜索：通配符模式搜索、按模式删除
- SQL + API 两种访问方式
- Bitcask 存储模型：内存仅索引（~20B/key），值留磁盘按需读取，内存占用可控
- 自动压缩：写入冗余超阈值时自动 Compact，回收磁盘空间

### 5.8 物化视图（P1）

- `CREATE/REFRESH/DROP MATERIALIZED VIEW`
- 增量捕获源表变更，增量更新统计表
- 支持增量聚合：SUM/COUNT/AVG/MIN/MAX
- 多维度 GROUP BY（time_bucket + 业务维度）
- 定时修正调度（cron）、修正时间窗口

### 5.9 ADO.NET Provider 与协议（P0）

- `DbProviderFactory`/`DbConnection`/`DbCommand`/`DbDataReader`/`DbParameter`/`DbTransaction`
- 参数化查询（防注入），命名参数 `@name`
- 连接字符串支持嵌入路径/服务器地址/只读标记
- `CommandTimeout`/`Cancel()`（尽力取消）
- TCP/IPv6 + 数据压缩

### 5.10 运维与可观测性

- **EXPLAIN**（P0）：访问对象、访问方式、预估行数、执行顺序
- **慢查询日志**（P0）：阈值可配置，记录 SQL/执行时间/扫描行数
- **监控指标**（P1）：QPS/延迟分位、IO 吞吐、网络吞吐、资源占用、业务指标
- **系统表**（P1）：表/索引/连接/事务列表、WAL/Binlog 状态
- **备份恢复**（P1）：拷贝目录 + Binlog 增量恢复

### 5.11 明确非目标（v1）

不支持：普通视图、触发器、存储过程、外键、CHECK、窗口函数、CTE、正则、全文检索、JSON 函数、自动水平分片、自动故障切换。

### 5.12 错误处理（P0）

- 语法错误/对象不存在/约束违反/只读拒绝/事务冲突/超时等有可区分的错误类型
- 同一类错误跨版本保持一致的错误码
- 函数参数非法或越界按 SQL 函数支持清单约定处理

### 5.13 安全（P2）

- 网络模式可支持认证（账号/Token）与权限控制
- TLS 传输加密作为扩展
- v1 不支持操作审计日志与静态数据加密

## 6. 性能与容量目标

### 6.1 性能基准（P0）

单节点嵌入模式基准：

| 指标 | 目标值 | 测试条件 |
|------|--------|---------|
| 点查询（主键） | P95 < 5ms | 百万行，热数据在内存 |
| 范围查询 | P95 < 50ms | 返回 100 行，有索引 |
| 单条插入 | > 10,000 QPS | WAL=NORMAL |
| 批量插入 | > 100,000 行/秒 | 1000 行/批 |
| 主键更新 | > 5,000 QPS | 单字段更新 |
| 主键删除 | > 5,000 QPS | WAL=NORMAL |
| JOIN 查询 | P95 < 200ms | 两表各 10 万行 |
| 聚合查询 | P95 < 500ms | 百万行 COUNT/SUM/AVG |

### 6.2 容量限制（P0）

| 限制项 | 建议值 | 硬限制 |
|--------|--------|--------|
| 单表行数 | < 10 亿 | 无逻辑上限 |
| 单行大小 | < 1MB | 16MB |
| 单表索引数 | < 10 | 32 |
| 索引键长度 | < 256 字节 | 4KB |
| 数据库表数量 | < 1000 | 无限制 |
| 数据库大小 | < 1TB | 无限制 |
| 并发连接数 | < 1000 | 10000 |
| 并发事务数 | < 500 | 无限制 |
| 向量维度 | < 2048 | 4096 |
| MQ 单表单日消息 | < 100 亿 | 无限制 |

### 6.3 资源占用（P1）

| 资源 | 基准值 |
|------|--------|
| 内存（空载） | < 100MB |
| CPU（空闲） | < 1% |
| CPU（高负载） | < 80%（1000 QPS 写入） |

## 7. 版本兼容性（P0）

### 7.1 数据文件

- 新版本**必须能打开**旧版本文件，首次写入自动升级
- 文件头包含版本号，不兼容版本明确拒绝
- 不支持降级

### 7.2 通信协议

- 连接时交换协议版本
- 同一大版本内保持兼容
- 跨大版本可不兼容，需升级指引

---

## 8. 集群与高可用（P1）

### 8.1 架构
- 1 主 + N 从（N ≤ 5），基于 Binlog 异步复制
- 应用层负责读写分离

### 8.2 一致性
- 最终一致性，可查询复制延迟

### 8.3 故障切换（P2）
- 手动提升从节点为主
- 不支持自动故障检测与切换

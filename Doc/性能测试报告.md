| 项目 | 详情 |

## 2. 测试概述

本报告使用 BenchmarkDotNet 对 NovaDb 的三大引擎（关系引擎、KV 引擎、时序引擎）在**嵌入模式**和**网络模式**下的核心操作进行基准测试，覆盖吞吐量（Op/s）和内存分配（Allocated）两个关键维度。

### 测试项目

| 引擎 | 模式 | 测试内容 |
|------|------|---------|
| SQL 关系引擎 | 嵌入模式 | INSERT / SELECT（主键/范围） / UPDATE / DELETE / 聚合查询 |
| KV 键值引擎 | 嵌入模式 | Set / Get / Set+Get / Delete / Exists / Inc / IncDouble / Search / SetAll / GetAll / SetExpiration / GetTtl / Add / Replace / GetString / GetAllKeys / GetExpiration / 海量写入10万条 |
| NovaCache 缓存层 | 嵌入模式 | Set\<String\> / Get\<String\> / Set\<Int32\> / Get\<Int32\> / Remove / ContainsKey / Increment / Decrement / SetExpire / GetExpire / Search / Set带TTL / Clear / Count / Keys / 通配符删除 / 海量写入10万条 |
| NovaCache 缓存层 | 网络模式 | Set\<String\> / Get\<String\> / Set\<Int32\> / Get\<Int32\> / Remove / ContainsKey / Increment / Decrement / SetExpire / GetExpire / Search / Set带TTL / Clear / Count / 海量写入1万条 |
| Flux 时序引擎 | 嵌入模式 | Append 单条 / AppendBatch 批量 / QueryRange 范围查询 |
| SQL 关系引擎 | 网络模式 | INSERT / SELECT（主键/范围） / UPDATE / 聚合查询 |

## 3. 嵌入模式 — SQL 关系引擎

测试 SQL 引擎在 `WalMode.None`（无 WAL，最高吞吐）和 `WalMode.Normal`（异步 WAL，平衡安全与性能）两种模式下的表现。

| 操作 | WAL 模式 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|---------|---------|------------|---------|
| Insert 单行插入 | None | 6.17 μs | **162,017** | 3.95 KB |
| Select 主键查询 | None | 337.47 μs | 2,963 | 589.30 KB |
| Select 范围查询 | None | 424.70 μs | 2,355 | 639.51 KB |
| Update 单行更新 | None | 1,048.69 μs | 954 | 590.53 KB |
| Delete + 重插 | None | 1,953.47 μs | 512 | 1,586.33 KB |
| 聚合 COUNT/AVG | None | 373.05 μs | 2,681 | 606.61 KB |
| Insert 单行插入 | Normal | 6.73 μs | **148,499** | 4.09 KB |
| Select 主键查询 | Normal | 341.25 μs | 2,930 | 589.30 KB |
| Select 范围查询 | Normal | 424.73 μs | 2,354 | 639.51 KB |
| Update 单行更新 | Normal | 1,070.15 μs | 934 | 590.67 KB |
| Delete + 重插 | Normal | 1,924.38 μs | 520 | 1,586.60 KB |
| 聚合 COUNT/AVG | Normal | 373.36 μs | 2,678 | 606.61 KB |

### 分析

- **INSERT 吞吐极高**：单行插入仅需 ~6 μs，WalMode.None 下达 **16.2 万 Op/s**，WalMode.Normal 下为 **14.8 万 Op/s**，开启异步 WAL 对写入性能的影响约 8.4%。
- **读操作稳定**：SELECT 主键查询和范围查询不受 WAL 模式影响，耗时一致（~337 μs / ~425 μs）。
- **内存分配**：INSERT 仅分配 ~4 KB，读/更新操作约 590-640 KB（包含 SQL 解析、AST 构建、结果集封装的完整链路）。

## 4. 嵌入模式 — KV 键值引擎

对比 64 字节和 1024 字节两种 Value 大小下的 KV 引擎性能。

### 基本读写操作

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Set 写入 | 64 B | 2,400.85 ns | **416,519** | 1,746 B |
| Get 读取 | 64 B | 25.83 ns | **38,713,209** | 0 B |
| Set+Get 混合 | 64 B | 2,392.44 ns | 417,983 | 1,128 B |
| Delete 删除 | 64 B | 3,133.33 ns | 319,149 | 1,904 B |
| Exists 存在检查 | 64 B | ~15 ns | ~66,000,000 | 0 B |
| Set 写入 | 1024 B | 7,389.83 ns | 135,321 | 6,658 B |
| Get 读取 | 1024 B | 25.98 ns | **38,495,538** | 0 B |
| Set+Get 混合 | 1024 B | 7,497.16 ns | 133,384 | 6,658 B |
| Delete 删除 | 1024 B | 7,941.37 ns | 125,923 | 6,816 B |
| Exists 存在检查 | 1024 B | ~15 ns | ~66,000,000 | 0 B |

### 原子操作与批量操作

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Inc 原子递增 | 64 B | ~3,200 ns | ~312,500 | ~2,100 B |
| IncDouble 浮点递增 | 64 B | ~3,300 ns | ~303,000 | ~2,100 B |
| SetAll 批量写入(10) | 64 B | ~18,500 ns | ~54,054 | ~17,800 B |
| GetAll 批量读取(10) | 64 B | ~500 ns | ~2,000,000 | ~600 B |
| Search 模式搜索 | 64 B | ~8,500 ns | ~117,647 | ~1,200 B |
| SetExpiration TTL设置 | 64 B | ~3,500 ns | ~285,714 | ~2,000 B |
| GetTtl TTL查询 | 64 B | ~15 ns | ~66,000,000 | 0 B |

### 分析

- **Get 读取极快**：纯内存 Dictionary 查找仅 ~26 ns，达到 **3,870 万 Op/s**，**零内存分配**。
- **Exists/GetTtl 更快**：纯内存索引查询仅 ~15 ns，达 **6,600 万 Op/s**，零分配。
- **Set 写入**：64B value 写入约 2.4 μs（含持久化日志），达 **41.6 万 Op/s**。1024B 随 value 增大，写入耗时和分配线性增长。
- **Inc 原子递增**：约 3.2 μs，含读-改-写全流程，达 **31.2 万 Op/s**。
- **批量操作**：SetAll 均摊耗时比单条 Set 节省约 23%，GetAll 均摊耗时仅 ~50 ns。
- **Value 大小影响**：从 64B 到 1024B（16 倍），写入耗时增长约 3 倍，分配增长约 3.8 倍，表明引擎在序列化和持久化上有较好的效率。
- **新增方法**：Add（仅新增）性能与 Set 接近；Replace 比 Set 多约 400 ns（需读旧值）；GetString 约 80 ns；GetAllKeys 约 30 μs（1000 条）；GetExpiration 零分配 ~15 ns。

### 海量数据性能（10 万条）

| 操作 | 平均耗时 | 均摊单条耗时 | 均摊吞吐 (Op/s) |
|------|---------|------------|----------------|
| 海量写入10万条(64B) | ~280 ms | ~2,800 ns | **~357,143** |
| 海量写入10万条(1024B) | ~820 ms | ~8,200 ns | ~121,951 |
| 海量批量写入10万条(SetAll, 64B) | ~220 ms | ~2,200 ns | **~454,545** |
| 海量数据搜索(10万条中搜索) | ~350 ms | — | — |

- **海量写入性能稳定**：10 万条 64B 均摊约 2.8 μs，与千条规模的 2.4 μs 相比衰减仅 17%。
- **批量写入优势明显**：SetAll 分批写入比逐条 Set 快约 21%。
- **读取不受规模影响**：10 万条数据中 Get 操作仍保持 ~26 ns。

> 详细分析参见 [KvStore 性能测试报告](性能/KvStore性能测试报告.md)

## 5. 嵌入模式 — NovaCache 缓存层

NovaCache 基于 KvStore 封装 ICache 接口，增加 `IPacketEncoder` 编码层（默认 `NovaJsonEncoder`），将应用类型序列化为二进制后交由 KvStore 存储。

### 字符串与整数操作

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Set\<String\> 写入 | 64 B | ~3,800 ns | ~263,158 | ~3,200 B |
| Get\<String\> 读取 | 64 B | ~350 ns | ~2,857,143 | ~400 B |
| Set+Get\<String\> 混合 | 64 B | ~4,200 ns | ~238,095 | ~3,600 B |
| Set\<Int32\> 写入 | 64 B | ~3,200 ns | ~312,500 | ~2,800 B |
| Get\<Int32\> 读取 | 64 B | ~250 ns | ~4,000,000 | ~300 B |
| ContainsKey 存在检查 | 64 B | ~15 ns | ~66,000,000 | 0 B |

### 原子操作与 TTL 管理

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Increment Int64 | 64 B | ~3,200 ns | ~312,500 | ~2,100 B |
| Increment Double | 64 B | ~3,300 ns | ~303,030 | ~2,100 B |
| Decrement Int64 | 64 B | ~3,200 ns | ~312,500 | ~2,100 B |
| SetExpire 设置过期 | 64 B | ~3,500 ns | ~285,714 | ~2,000 B |
| GetExpire 获取过期 | 64 B | ~15 ns | ~66,000,000 | 0 B |
| Search 模式搜索 | 64 B | ~8,500 ns | ~117,647 | ~1,200 B |
| Remove 删除 | 64 B | ~5,000 ns | ~200,000 | ~4,500 B |
| Set 带TTL写入 | 64 B | ~3,900 ns | ~256,410 | ~3,300 B |

### NovaCache vs KvStore 性能对比

| 操作类型 | NovaCache 额外开销 | 说明 |
|---------|-------------------|------|
| 纯代理操作（ContainsKey/Increment/GetExpire） | **0%** | 直接转发到 KvStore |
| 写入操作 | **22-37%** | Encoder.Encode + ReadBytes 拷贝 |
| 读取操作 | 绝对值 +324 ns | Encoder.Decode 反序列化（仍在亚微秒级） |

### 分析

- **纯代理操作零开销**：ContainsKey、Increment、GetExpire 直接转发到 KvStore，性能完全一致。
- **写入增加 22-37% 编码开销**：编码层的主要开销来自 `Encoder.Encode()` + `ReadBytes()` 拷贝。
- **读取增加反序列化开销**：虽然倍率看似较高（13-23×），但绝对耗时仍在 250-600 ns（亚微秒级），满足高性能缓存要求。
- **整数类型更高效**：Int32 编解码比 String 更轻量，适合计数器和标志位场景。
- **新增操作**：Clear 约 50 μs、Count 零分配 ~15 ns、Keys 约 30 μs、通配符删除约 15 μs。

### 嵌入模式海量数据性能（10 万条）

| 操作 | 平均耗时 | 均摊单条耗时 | 均摊吞吐 (Op/s) |
|------|---------|------------|----------------|
| 嵌入模式海量写入10万条(64B) | ~420 ms | ~4,200 ns | **~238,095** |
| 嵌入模式海量写入10万条(1024B) | ~1,050 ms | ~10,500 ns | ~95,238 |

- 编码层对海量写入的开销比例与单条写入一致（+50% for 64B, +28% for 1024B）。
- 10 万条写入性能衰减约 10%，表明缓存层在大规模数据下扩展良好。

> 详细分析参见 [NovaCache 嵌入模式性能测试报告](性能/NovaCache嵌入模式性能测试报告.md)

## 5.1 网络模式 — NovaCache 缓存层

NovaCache 网络模式通过 TCP RPC（NovaClient → NovaServer → KvStore）进行远程缓存操作。

### 字符串与整数操作

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Net Set\<String\> 写入 | 64 B | ~150,000 ns | ~6,667 | ~12,000 B |
| Net Get\<String\> 读取 | 64 B | ~120,000 ns | ~8,333 | ~10,000 B |
| Net Set\<Int32\> 写入 | 64 B | ~145,000 ns | ~6,897 | ~11,000 B |
| Net Get\<Int32\> 读取 | 64 B | ~115,000 ns | ~8,696 | ~9,500 B |
| Net ContainsKey 存在检查 | 64 B | ~110,000 ns | ~9,091 | ~8,000 B |

### 原子操作与 TTL 管理

| 操作 | Value 大小 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|-----------|---------|------------|---------|
| Net Increment Int64 | 64 B | ~130,000 ns | ~7,692 | ~10,000 B |
| Net SetExpire 设置过期 | 64 B | ~130,000 ns | ~7,692 | ~10,000 B |
| Net GetExpire 获取过期 | 64 B | ~110,000 ns | ~9,091 | ~8,000 B |
| Net Search 模式搜索 | 64 B | ~140,000 ns | ~7,143 | ~12,000 B |

### 嵌入模式 vs 网络模式对比

| 操作 | 嵌入模式 | 网络模式 | 倍率 |
|------|---------|---------|------|
| Set\<String\> (64B) | ~3,800 ns | ~150,000 ns | 39× |
| Get\<String\> (64B) | ~350 ns | ~120,000 ns | 343× |
| ContainsKey | ~15 ns | ~110,000 ns | 7,260× |
| Increment Int64 | ~3,200 ns | ~130,000 ns | 41× |
| Search | ~8,500 ns | ~140,000 ns | 16× |

### 网络模式海量数据性能

| 操作 | 平均耗时 | 均摊单条耗时 | 均摊吞吐 (Op/s) |
|------|---------|------------|----------------|
| 网络模式海量写入1万条(64B) | ~1.5 s | ~150,000 ns | ~6,667 |
| 网络模式海量写入后读取1万条(64B) | ~2.7 s | 写 ~150 μs + 读 ~120 μs | — |

### 分析

- **网络开销主导**：每次 RPC 固定约 110-150 μs 的网络延迟（TCP 本地回环），远大于引擎操作本身。
- **嵌入模式越快的操作，网络模式的倍率差越大**：ContainsKey 差 7,260 倍（因为原操作仅 15 ns），而 Set 差 39 倍（因为原操作含磁盘 IO）。
- **网络模式适用场景**：多客户端并发、跨进程共享、集群复制的生产环境。嵌入模式适合单进程高性能场景。
- **海量数据**：网络模式逐条 RPC 无批量优化，1 万条写入约 1.5 秒，适合中等规模数据操作。

> 详细分析参见 [NovaCache 网络模式性能测试报告](性能/NovaCache网络模式性能测试报告.md)

## 6. 嵌入模式 — Flux 时序引擎

| 操作 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|---------|------------|---------|
| Append 单条写入 | 261.23 μs | **3,828** | 1.88 KB |
| AppendBatch 批量写入 (100 条) | 55,041.81 μs | 18 | 208.29 KB |
| QueryRange 时间范围查询 | 2.67 μs | **374,334** | 2.21 KB |

### 分析

- **范围查询极快**：2.67 μs 即可完成时间范围查询，达 **37.4 万 Op/s**，内存仅 2.21 KB。
- **单条写入**：由于时序引擎包含分区键计算、消息 ID 自增、持久化等逻辑，单条写入约 261 μs。
- **批量写入**：100 条批量写入约 55 ms（含持久化），单条均摊约 550 μs。随着数据累积，持久化开销线性增长（测试过程中数据量不断增大）。
- **内存效率**：单条写入仅 1.88 KB，批量写入每条均摊约 2.08 KB，分配比例稳定。

## 7. 网络模式 — SQL 关系引擎

通过 TCP 网络（NovaServer + ApiServer）进行远程 SQL 执行，对比嵌入模式的额外网络开销。

| 操作 | 平均耗时 | 吞吐 (Op/s) | 内存分配 |
|------|---------|------------|---------|
| Server Insert 插入 | 154.53 μs | **6,471** | 11.41 KB |
| Server Select 主键查询 | 508.32 μs | 1,967 | 601.57 KB |
| Server Select 范围查询 | 886.77 μs | 1,128 | 999.66 KB |
| Server Update 更新 | 1,239.66 μs | 807 | 597.88 KB |
| Server 聚合查询 | 540.02 μs | 1,852 | 617.93 KB |

### 嵌入 vs 网络模式对比

| 操作 | 嵌入模式 (WalNone) | 网络模式 | 倍率 |
|------|-------------------|---------|------|
| Insert | 6.17 μs | 154.53 μs | 25.0× |
| Select 主键 | 337.47 μs | 508.32 μs | 1.5× |
| Select 范围 | 424.70 μs | 886.77 μs | 2.1× |
| Update | 1,048.69 μs | 1,239.66 μs | 1.2× |
| 聚合查询 | 373.05 μs | 540.02 μs | 1.4× |

### 分析

- **Insert 差距最大**（25×）：嵌入模式 INSERT 仅 6 μs，而网络模式需经过序列化 → TCP 传输 → 反序列化 → 执行 → 返回，额外约 148 μs。
- **读操作差距较小**（1.2-2.1×）：原始查询本身耗时较长（数百 μs），网络开销占比降低。
- **内存分配**：网络模式因网络缓冲区和协议封装，每次操作额外分配约 7-360 KB。
- **网络模式适用场景**：多客户端并发访问、跨进程数据共享、需要集群复制的生产环境。

## 10. 注意事项

1. **测试环境差异**：CI 环境的性能数据仅供参考，实际生产环境的性能取决于硬件配置、磁盘类型、并发级别等因素。
2. **数据规模**：基本测试使用 1,000 条预置数据，海量数据测试覆盖 10 万条（KV/NovaCache 嵌入模式）和 1 万条（NovaCache 网络模式），大数据量场景下性能特征可能不同。
3. **InProcess 模式**：为兼容项目构建配置，基准测试使用 InProcess 工具链，可能存在微小的测量偏差。
4. **网络模式端口**：网络模式测试使用随机端口，避免与本地服务冲突。
5. **详细报告**：各引擎的详细性能分析请参见 `Doc/性能/` 目录下的独立报告：
   - [KvStore 性能测试报告](性能/KvStore性能测试报告.md)
   - [NovaCache 嵌入模式性能测试报告](性能/NovaCache嵌入模式性能测试报告.md)
   - [NovaCache 网络模式性能测试报告](性能/NovaCache网络模式性能测试报告.md)

## 1. 系统架构概览

NewLife.NovaDb（诺娃数据库）是一个纯 C# 实现的混合数据库引擎，支持 .NET Framework 4.5+（含 .NET Core / .NET 5~10），支持嵌入式/服务器双模部署。融合关系型、时序、消息队列、KV 能力。

### 1.1 核心特性

- **纯 C# 实现**：无 native 依赖，跨平台一致
- **双模部署**：嵌入式（进程内）/ 服务器模式（独立进程）
- **混合引擎**：Nova（关系型）+ Flux（时序 + MQ）+ KV（键值）
- **MVCC 事务**：Read Committed 隔离级别，乐观并发
- **冷热分离**：热数据内存索引，冷数据 MMF 按需加载
- **双日志**：WAL（崩溃恢复）+ Binlog（主从复制）

### 1.2 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                      应用层 (Application)                    │
├─────────────────────────────────────────────────────────────┤
│                   ADO.NET Provider (Client)                  │
│  NovaClientFactory │ NovaConnection │ NovaCommand            │
│  NovaDataReader │ NovaParameter │ NovaTransaction            │
├─────────────────────────────────────────────────────────────┤
│                      SQL 引擎 (Sql)                          │
│        SqlLexer → SqlParser → SqlAst → SqlEngine             │
├──────────┬──────────┬──────────┬────────────────────────────┤
│ Nova     │ Flux     │ KV       │ 服务器 (Server)             │
│ Engine   │ Engine   │ Store    │ NovaServer │ NovaController  │
│ (关系型)  │ (时序+MQ) │ (键值)    │ NovaSession                │
├──────────┴──────────┴──────────┼────────────────────────────┤
│       事务层 (Tx) — MVCC       │     集群 (Cluster)          │
│  TransactionManager │ RowVersion│ ReplicationManager          │
├────────────────────────────────┴────────────────────────────┤
│               WAL 日志层 (WAL + Binlog)                      │
│  WalWriter │ WalRecovery │ BinlogWriter │ BinlogRecord       │
├─────────────────────────────────────────────────────────────┤
│                    存储层 (Storage)                           │
│  MmfPager │ PageCache │ FileHeader │ TableFileManager         │
├─────────────────────────────────────────────────────────────┤
│                    核心层 (Core)                              │
│  DbOptions │ DataType │ IDataCodec │ NovaException            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 模块依赖

```
Client → Sql → Engine → Tx → WAL → Storage → Core
Server → Engine → Tx → WAL → Storage → Core
Cluster → Binlog → WAL → Storage → Core
```

## 3. 设计决策记录（DDR）

### D1: 纯托管实现
不引入 native 依赖，保持跨平台一致性。

### D2: MMF 作为主要存储
利用操作系统虚拟内存管理，64 位系统 128TB 虚拟地址空间，OS 按需分页映射。

### D3: SkipList 而非 B+ 树
实现简单、并发友好，适合内存索引场景。

### D4: Read Committed 隔离级别
平衡一致性与并发性能，满足大多数业务场景。

### D5: 表级 WAL
每表独立 WAL，减少并发竞争，简化恢复。

### D6: 冷热分离索引
千万行表仅查热点时内存 < 20MB，冷数据自动卸载至 MMF。

### D7: SQL 子集策略
覆盖 ~60% SQL92 标准，聚焦业务常用场景，不追求完整性。

### D8: Nova 不分片
单表单文件，MMF 高效处理超大文件。分片路由复杂度大于收益，如需分表由应用层完成。

### D9: 表级并发控制
嵌入模式表级跨进程互斥"一写多读"；服务器模式单进程 MVCC 行级并发。

### D10: nova.db 二进制元数据
独立元数据存储，DDL 原子写入/替换，保证崩溃安全。

### D11: 嵌入与服务器存储统一
两种模式相同文件布局，目录可直接互用。

### D12: KV 多表
一库多 KV 表（`ENGINE=KV`），每表独立默认 TTL。

### D13: MQ 消息即时序行
Flux 中 MQ 消息 = 时序表行（MessageId + Fields），统一存储模型。

### D14: Binlog 独立于 WAL
WAL 是表级物理日志（崩溃恢复），Binlog 是库级逻辑日志（复制/审计），职责正交。

### D15: Binlog 可手工删除
文件自包含无交叉依赖，删除不导致崩溃；代价是落后的从节点需全量重建。

### D16: MQ 功能融合到 NovaQueue
废弃 StreamManager 中间层，消费组管理、延迟消息、死信队列、阻塞读取等 Stream/MQ 语义全部融合到 `NovaQueue<T>`。NovaQueue 直接持有 FluxEngine，减少一层间接调用，简化依赖链。

## 1. 系统架构概览

NewLife.NovaDb（诺娃数据库）是一个纯 C# 实现的混合数据库引擎，支持 .NET Framework 4.5+（含 .NET Core / .NET 5~10），支持嵌入式/网络双模部署。融合关系型、时序、消息队列、KV 能力。

### 1.1 核心特性

- **纯 C# 实现**：无 native 依赖，跨平台一致
- **双模部署**：嵌入模式（进程内）/ 网络模式（独立进程）
- **混合引擎**：Nova（关系型）+ Flux（时序 + MQ）+ KV（键值）
- **MVCC 事务**：Read Committed 隔离级别，乐观并发
- **冷热分离**：热数据内存索引，冷数据 MMF 按需加载
- **双日志**：WAL（崩溃恢复）+ Binlog（主从复制）

## 2. 统一入口与四大场景

### 2.1 核心概念

**连接字符串 = 数据库**。一个连接字符串对应一个数据库，数据库内可同时包含：
- **多张关系表**（Nova 引擎）
- **多张时序表**（Flux 引擎，同时作为 MQ 的底层存储）
- **多个 KV 库**（KvStore 引擎，按表名隔离）

四大场景共用同一个连接字符串，共用同一数据库目录（嵌入模式）或同一服务端连接（网络模式）。

### 2.2 统一入口 `NovaClientFactory`

`NovaClientFactory` 继承 `DbProviderFactory`，是所有场景的统一入口：

```
NovaClientFactory (统一入口，单例 Instance)
│
├── CreateConnection()           → NovaConnection (ADO.NET)
│   用于：关系型 SQL、时序 SQL、DDL
│
├── GetCacheProvider(connStr)    → NovaCacheProvider (ICacheProvider)
│   用于：同时需要 KV 缓存 + MQ 队列的场景
│   └── .Cache                  → ICache (默认 KV 表)
│   └── .GetCache(tableName)    → ICache (指定 KV 表)
│   └── .GetQueue<T>(topic)     → IProducerConsumer<T> (MQ 队列)
│
├── GetCache(connStr, tableName) → ICache (NovaCache)
│   用于：仅需 KV 缓存，直接获取指定表
│
└── GetQueue<T>(connStr, topic)  → IProducerConsumer<T> (NovaQueue)
    用于：仅需 MQ 队列，直接获取指定 topic
```

### 2.3 双模式架构

#### 2.3.1 嵌入模式

连接字符串：`Data Source=../data/mydb`

```
NovaClientFactory
  └── EmbeddedDatabase (按数据库目录缓存，同一目录共用实例)
        ├── SqlEngine           → 关系型/时序 SQL 操作
        ├── KvStore × N         → 多个 KV 表 (kv_default, kv_session, ...)
        └── FluxEngine          → 时序表 + MQ 队列 (共用引擎)
```

- 同一数据库目录只创建一个 `EmbeddedDatabase` 实例，避免引擎重复创建
- `NovaConnection.Open()` 时通过 `EmbeddedDatabase` 获取 `SqlEngine`
- `GetCache()` 通过 `EmbeddedDatabase.GetKvStore(tableName)` 获取 KV 引擎
- `GetQueue()` 通过 `EmbeddedDatabase.FluxEngine` 获取 Flux 引擎
- 数据库目录结构：

```
mydb/                       ← 数据库目录 (连接字符串指向此处)
├── nova.db                  ← 元数据文件 (表定义)
├── users.nvd / .nvi         ← 关系表 users 数据/索引
├── orders.nvd / .nvi        ← 关系表 orders 数据/索引
├── flux/                    ← Flux 引擎目录 (时序 + MQ)
│   ├── metrics.fxd          ← 时序表 metrics
│   └── events.fxd           ← MQ topic: events
├── kv_default/              ← KV 表 "default"
├── kv_session/              ← KV 表 "session"
└── wal/                     ← WAL 日志
```

#### 2.3.2 网络模式

连接字符串：`Server=localhost;Port=3306;Database=mydb`

```
NovaClientFactory
  └── NovaPoolManager
        └── NovaClientPool (按连接字符串缓存，共用连接池)
              └── NovaClient × N (RPC 客户端，从池中借用/归还)
                    ├── Nova/* → SQL 操作 (Execute/Query/Transaction)
                    ├── Kv/*   → KV 操作 (Set/Get/Delete/Exists)
                    └── Flux/* → MQ 操作 (Publish/ReadGroup/Ack)
```

- 同一连接字符串共用一个 `NovaClientPool`，复用 TCP 连接
- `NovaConnection.Open()` 从连接池借用 `NovaClient`，`Close()` 时归还
- `GetCache()` 从连接池借用 `NovaClient`，封装为 `NovaCache`
- `GetQueue()` 从连接池借用 `NovaClient`，封装为 `NovaQueue`
- 四大场景共用同一个连接池，无需为不同场景建立独立连接

### 2.4 四大场景使用方式

#### 场景 1：关系型（Nova 引擎）— ADO.NET SQL

```csharp
// 通过 ADO.NET 标准接口操作关系表，嵌入/网络模式仅连接字符串不同
var factory = NovaClientFactory.Instance;
using var conn = (NovaConnection)factory.CreateConnection();
conn.ConnectionString = "Data Source=../data/mydb";  // 嵌入模式
// conn.ConnectionString = "Server=host;Port=3306;Database=mydb";  // 网络模式
conn.Open();

// DDL
conn.ExecuteNonQuery("CREATE TABLE users (id INT PRIMARY KEY, name STRING(50), age INT)");

// DML
using var cmd = conn.CreateCommand();
cmd.CommandText = "INSERT INTO users (id, name, age) VALUES (@id, @name, @age)";
cmd.Parameters.Add(new NovaParameter("@id", 1));
cmd.Parameters.Add(new NovaParameter("@name", "Alice"));
cmd.Parameters.Add(new NovaParameter("@age", 25));
cmd.ExecuteNonQuery();

// 查询
cmd.CommandText = "SELECT * FROM users WHERE age > @minAge";
cmd.Parameters.Clear();
cmd.Parameters.Add(new NovaParameter("@minAge", 18));
using var reader = cmd.ExecuteReader();
```

#### 场景 2：时序库（Flux 引擎）— ADO.NET SQL

```csharp
// 时序表通过 ENGINE=FLUX 创建，同样使用 ADO.NET SQL 操作
using var conn = new NovaConnection("Data Source=../data/mydb");
conn.Open();

// 创建时序表
conn.ExecuteNonQuery(@"CREATE TABLE metrics (
    time DATETIME, device STRING(32), temperature DOUBLE, humidity DOUBLE
) ENGINE=FLUX");

// 写入时序数据
conn.ExecuteNonQuery(@"INSERT INTO metrics (time, device, temperature, humidity)
    VALUES (NOW(), 'sensor-01', 23.5, 65.2)");

// 时间范围聚合查询
using var cmd = conn.CreateCommand();
cmd.CommandText = @"SELECT device, AVG(temperature), MAX(humidity)
    FROM metrics WHERE time > @start GROUP BY device";
cmd.Parameters.Add(new NovaParameter("@start", DateTime.Now.AddHours(-1)));
using var reader = cmd.ExecuteReader();
```

#### 场景 3：KV 缓存 — ICache API

```csharp
var factory = NovaClientFactory.Instance;
var connStr = "Data Source=../data/mydb";  // 或网络模式连接字符串

// 方式 1：直接获取指定 KV 表
ICache cache = factory.GetCache(connStr, "session");
cache.Set("user:1001", new UserInfo { Name = "Alice" }, 3600);
var user = cache.Get<UserInfo>("user:1001");

// 方式 2：通过 CacheProvider 获取多个 KV 表
var provider = factory.GetCacheProvider(connStr);
ICache defaultCache = provider.Cache;                  // 默认 KV 表
ICache sessionCache = provider.GetCache("session");    // session 表
ICache configCache = provider.GetCache("config");      // config 表
```

#### 场景 4：MQ 消息队列 — IProducerConsumer API

```csharp
var factory = NovaClientFactory.Instance;
var connStr = "Data Source=../data/mydb";

// 方式 1：直接获取指定 topic 的队列
var queue = factory.GetQueue<OrderEvent>(connStr, "orders", "order-processor");
queue.Add(new OrderEvent { OrderId = 1001, Action = "created" });

// 方式 2：通过 CacheProvider 获取队列
var provider = factory.GetCacheProvider(connStr);
var eventQueue = provider.GetQueue<OrderEvent>("orders", "order-processor");
```

#### 综合场景：四合一

```csharp
// 同一连接字符串覆盖四大场景
var connStr = "Data Source=../data/mydb";
var factory = NovaClientFactory.Instance;

// ① 关系型 — ADO.NET
using var conn = new NovaConnection(connStr);
conn.Open();
conn.ExecuteNonQuery("INSERT INTO users (id, name) VALUES (1, 'Alice')");

// ② 时序库 — ADO.NET
conn.ExecuteNonQuery("INSERT INTO metrics (time, device, value) VALUES (NOW(), 's1', 23.5)");

// ③ KV 缓存 — ICache
var cache = factory.GetCache(connStr, "session");
cache.Set("token:abc", "user-1001", 3600);

// ④ MQ 队列 — IProducerConsumer
var queue = factory.GetQueue<String>(connStr, "notifications", "push-service");
queue.Add("New order #1001");
```

### 2.6 关键设计决策

| 决策 | 说明 |
|------|------|
| **连接字符串即数据库** | 一个连接字符串对应一个数据库目录/远程库，内含所有引擎数据 |
| **嵌入模式引擎共用** | 同一目录的 EmbeddedDatabase 全局缓存，避免多次创建引擎 |
| **网络模式连接池共用** | 四大场景共用 NovaClientPool，NovaClient 支持 SQL/KV/MQ 全部 RPC |
| **关系型/时序走 ADO.NET** | SQL 操作通过 NovaConnection → SqlEngine/NovaClient，标准接口 |
| **KV 走 ICache** | NovaCache 封装 KvStore（嵌入）或 NovaClient.Kv*（服务器） |
| **MQ 走 IProducerConsumer** | NovaQueue 封装 FluxEngine（嵌入）或 NovaClient.Flux*（服务器） |
| **NovaCacheProvider 聚合** | 同时提供 ICache + IProducerConsumer，适合同时需要缓存和队列的场景 |

## 4. 设计决策记录（DDR）

### D1: 纯托管实现
不引入 native 依赖，保持跨平台一致性。

### D2: MMF 作为主要存储
利用操作系统虚拟内存管理，64 位系统 128TB 虚拟地址空间，OS 按需分页映射。

### D3: SkipList 而非 B+ 树
实现简单、并发友好，适合内存索引场景。

### D4: Read Committed 隔离级别
平衡一致性与并发性能，满足大多数业务场景。

### D5: 表级 WAL
每表独立 WAL，减少并发竞争，简化恢复。

### D6: 冷热分离索引
千万行表仅查热点时内存 < 20MB，冷数据自动卸载至 MMF。

### D7: SQL 子集策略
覆盖 ~60% SQL92 标准，聚焦业务常用场景，不追求完整性。

### D8: Nova 不分片
单表单文件，MMF 高效处理超大文件。分片路由复杂度大于收益，如需分表由应用层完成。

### D9: 表级并发控制
嵌入模式表级跨进程互斥"一写多读"；网络模式单进程 MVCC 行级并发。

### D10: nova.db 二进制元数据
独立元数据存储，DDL 原子写入/替换，保证崩溃安全。

### D11: 嵌入与网络存储统一
两种模式相同文件布局，目录可直接互用。

### D12: KV 多表
一库多 KV 表（`ENGINE=KV`），每表独立默认 TTL。采用 Bitcask 存储模型，内存仅存索引（~20B/key），值留磁盘按需读取，百万键仅占 ~20MB 内存。

### D13: MQ 消息即时序行
Flux 中 MQ 消息 = 时序表行（MessageId + Fields），统一存储模型。

### D14: Binlog 独立于 WAL
WAL 是表级物理日志（崩溃恢复），Binlog 是库级逻辑日志（复制/审计），职责正交。

### D15: Binlog 可手工删除
文件自包含无交叉依赖，删除不导致崩溃；代价是落后的从节点需全量重建。

### D16: MQ 功能融合到 NovaQueue
废弃 StreamManager 中间层，消费组管理、延迟消息、死信队列、阻塞读取等 Stream/MQ 语义全部融合到 `NovaQueue<T>`。NovaQueue 直接持有 FluxEngine，减少一层间接调用，简化依赖链。

### D17: NovaClientFactory 统一入口
`NovaClientFactory` 作为四大场景（关系型、时序、MQ、KV）的统一入口。一个连接字符串对应一个数据库，数据库内可包含多张关系表、多张时序表（即多个 MQ topic）、多个 KV 库。嵌入模式下通过 `EmbeddedDatabase` 共用引擎实例，网络模式下通过 `NovaClientPool` 共用连接池。四大场景共用同一个 `NovaClient`，避免为不同场景建立独立连接。

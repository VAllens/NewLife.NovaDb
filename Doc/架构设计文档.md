# NewLife.NovaDb 架构设计文档

## 1. 系统架构概览

NewLife.NovaDb（诺娃数据库）是一个纯 C# 实现的混合数据库引擎，支持 .NET Framework 4.5+ 平台（含 .NET Core / .NET 5~10），支持嵌入式/服务器双模部署。融合关系型、时序、消息队列、NoSQL(KV) 能力。

### 1.1 整体架构图

```
┌──────────────────────────────────────────────────────────────────────┐
│                         应用层 (Application)                         │
├──────────────────────────────────────────────────────────────────────┤
│                     ADO.NET Provider (Client)                        │
│  NovaClientFactory │ NovaConnectionStringBuilder │ NovaDataAdapter   │
│  NovaConnection │ NovaCommand │ NovaDataReader │ NovaParameter       │
├──────────────────────────────────────────────────────────────────────┤
│                      SQL 引擎 (Sql)                                  │
│      SqlLexer → SqlParser → SqlAst → SqlEngine                      │
├─────────┬────────────┬──────────────┬────────────────────────────────┤
│ Nova    │ Flux       │ KV           │ 服务器 (Server)                 │
│ Engine  │ Engine     │ Store        │ NovaServer │ NovaProtocol   │
│ (关系型) │ (时序+MQ)   │ (键值)        │ NovaController│ NovaSession   │
├─────────┴────────────┴──────────────┼────────────────────────────────┤
│          事务层 (Tx)                 │       集群 (Cluster)            │
│ TransactionManager │ Transaction    │ ReplicationManager              │
│ RowVersion │ MVCC                   │ ReplicaClient │ NodeInfo        │
├─────────────────────────────────────┴────────────────────────────────┤
│                      WAL 日志层 (WAL + Binlog)                       │
│  WalWriter │ WalRecord │ WalRecovery │ BinlogWriter │ BinlogRecord   │
├──────────────────────────────────────────────────────────────────────┤
│                      存储层 (Storage)                                │
│  MmfPager │ PageCache │ FileHeader │ PageHeader │ TableFileManager    │
├──────────────────────────────────────────────────────────────────────┤
│                      核心层 (Core)                                   │
│  DbOptions │ DataType │ IDataCodec │ NovaException │ ErrorCode      │
└──────────────────────────────────────────────────────────────────────┘
```

### 1.2 模块依赖关系

```
Client → Sql → Engine → Tx → WAL → Storage → Core
Server → Engine → Tx → WAL → Storage → Core
Cluster → Binlog → WAL → Storage → Core
```

## 2. 核心层 (Core)

### 2.1 数据类型系统 (`DataType`)

支持 8 种基础数据类型，严格映射 C# 类型：

| DataType | C# 类型 | 存储大小 |
|----------|---------|---------|
| Boolean | Boolean | 1 字节 |
| Int32 | Int32 | 4 字节 |
| Int64 | Int64 | 8 字节 |
| Double | Double | 8 字节 |
| Decimal | Decimal | 16 字节 |
| String | String | 变长 (UTF-8) |
| ByteArray | Byte[] | 变长 |
| DateTime | DateTime | 8 字节 (Ticks) |

### 2.2 数据编解码 (`IDataCodec`)

- `DefaultDataCodec` 实现二进制编解码
- NULL 值编码为 -1 长度前缀
- 字符串使用 UTF-8 编码，带 4 字节长度前缀
- Decimal 使用 128 位 (16 字节) 表示

### 2.3 异常与错误码 (`NovaException`)

统一异常类型，按模块分段编码：
- 1000-1999: 文件/存储错误
- 2000-2999: 解析/语法错误
- 3000-3999: 事务错误
- 4000-4999: 表/分片/流/KV 错误
- 5000-5999: 不支持的操作
- 6000-6999: I/O 错误
- 7000-7999: 连接/协议错误
- 8000-8999: 复制/集群错误

### 2.4 数据库选项 (`DbOptions`)

嵌入模式与服务模式共用的基础配置：

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| Path | - | 数据库文件夹路径 |
| WalMode | Normal | WAL 模式 (Full/Normal/None) |
| PageSize | 4096 | 页面大小 (字节) |
| PageCacheSize | 1024 | 页面缓存大小 |
| HotWindowSeconds | 600 | 热数据窗口 (秒) |
| ColdEvictionSeconds | 1800 | 冷数据淘汰时间 (秒) |
| EnableChecksum | true | 是否启用校验和验证 |
| ReadOnly | false | 是否只读模式（禁止写操作，跳过 WAL 初始化，提升读取性能） |

### 2.5 服务模式选项 (`ServerDbOptions` 继承 `DbOptions`)

服务模式专属配置，包含 Binlog 相关设置：

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| EnableBinlog | true | 是否启用 Binlog（服务模式默认启用） |
| BinlogMaxFileSize | 256MB | 单个 Binlog 文件最大大小，达到后滚动到新文件 |
| BinlogRetentionDays | 0 | Binlog 保留天数，0 表示不按时间清理 |
| BinlogMaxTotalSize | 0 | Binlog 总磁盘空间上限（字节），0 表示不限制 |

## 3. 存储层 (Storage)

### 3.1 文件布局

```
{DatabasePath}/                          # 文件夹即数据库
├── nova.db                              # 数据库元数据（二进制：表/列/索引定义）
│
│   ── Binlog 日志（主从同步/增量备份）──
├── nova.000001.binlog                   # 已滚动的 Binlog（从节点确认后可清理）
├── nova.000002.binlog                   # 当前活跃的 Binlog（正在追加写入）
│
│   ── Nova Engine 关系表 ──
├── {TableName}.nvd                      # 表数据文件（单文件，MMF 访问）
├── {TableName}.nvi                      # 表主键索引文件
├── {TableName}_{IndexName}.nvi          # 表二级索引文件（每索引独立）
├── {TableName}.wal                      # 表 WAL 日志（写入时存在，checkpoint 后删除）
├── {TableName}.lock                     # 表级写锁（嵌入模式，写入时存在）
│
│   ── Flux Engine 时序/MQ 表 ──
├── {StreamName}.nvd                     # 时序表当前分片
├── {StreamName}.nvi                     # 时序表时间主键索引
├── {StreamName}_{yyyyMMdd}.nvd          # 时序表按天历史分片
│
│   ── KV Engine 键值表 ──
├── {KvTableName}.nvd                    # KV 表数据文件
└── {KvTableName}.nvi                    # KV 表主键索引（Key）
```

**说明**：
- 每表文件组使用相同的表名前缀，通过后缀区分类型（`.nvd`/`.wal`/`.nvi`/`.lock`）
- 二级索引文件命名为 `{TableName}_{IndexName}.nvi`
- Binlog 为数据库级逻辑日志（`nova.*.binlog`），记录已提交事务的逻辑操作，用于主从复制和增量备份
- Nova Engine 不分片，每表一个 `.nvd` 文件（MMF 支持超大文件）
- Flux Engine 按时间分片，分片文件命名为 `{StreamName}_{yyyyMMdd}.nvd`
- 所有文件平铺在数据库目录下，无需为每个表创建独立目录

### 3.2 文件头 (`FileHeader`)

32 字节固定大小，所有 `.nvd`/`.nvi`/`.wal`/`.binlog` 和 `nova.db` 文件共享此格式：
- Magic Number: `0x4E4F5641` ("NOVA") - 4 字节
- Version: 文件格式版本 - 1 字节
- FileType: 文件类型 - 1 字节
  - `1` = Data (.nvd 数据文件)
  - `2` = Index (.nvi 索引文件)
  - `3` = Wal (.wal WAL 日志文件)
  - `4` = Binlog (主从同步/增量备份的逻辑日志文件)
  - `5` = Metadata (nova.db 数据库元数据文件)
- PageSizeShift: 页大小位移 (实际页大小 = 1 << shift) - 1 字节
- Flags: 特性标志 (bit0=加密, bit1=压缩, bit2=只读) - 1 字节
- CreateTime: 创建时间 (UTC 毫秒) - 8 字节
- Checksum: CRC32 校验和 (覆盖 bytes[0-15]) - 4 字节
- Reserved: 预留扩展 - 12 字节

### 3.2.1 数据库元数据 (`nova.db`)

`nova.db` 是数据库的核心元数据文件，二进制格式，包含整个数据库的完整定义：

```
nova.db 结构：
├── FileHeader (32B)                     # 标准文件头，FileType=Metadata
├── DatabaseInfo                         # 数据库信息
│   ├── DatabaseName (string)            # 数据库名称
│   ├── CreateTime (datetime)            # 创建时间
│   └── ModifyTime (datetime)            # 最后修改时间
└── TableCatalog[]                       # 表目录（所有表的定义）
    ├── TableName (string)               # 表名
    ├── EngineType (byte)                # 引擎类型：0=Nova, 1=Flux, 2=KV
    ├── Comment (string)                 # 表注释
    ├── CreateTime (datetime)            # 建表时间
    ├── ColumnCount (int)                # 列数
    ├── Columns[]                        # 列定义列表
    │   ├── Name (string)                # 列名
    │   ├── DataType (byte)              # 数据类型
    │   ├── Length (int)                 # 长度（STRING/BINARY）
    │   ├── IsNullable (bool)            # 是否可空
    │   ├── IsPrimaryKey (bool)          # 是否主键
    │   ├── IsAutoIncrement (bool)       # 是否自增
    │   ├── DefaultValue (variant)       # 默认值
    │   └── Comment (string)             # 列注释
    ├── IndexCount (int)                 # 索引数
    └── Indexes[]                        # 索引定义列表
        ├── Name (string)                # 索引名
        ├── IsUnique (bool)              # 是否唯一
        └── ColumnNames[] (string)       # 包含的列名列表
```

**修改机制**：DDL 操作（CREATE/ALTER/DROP TABLE）时，先写临时文件 `nova.db.tmp`，完成后原子重命名为 `nova.db`，保证崩溃安全。

**启动加载**：引擎启动时一次性读取 `nova.db`，在内存中构建表 Schema 缓存。取代 `_sys_tables` 系统表概念，简化引导流程。

### 3.3 页面管理

- **PageHeader**: 32 字节页头，包含 PageId、Type、LSN、Checksum
- **MmfPager**: 基于 MemoryMappedFile 的分页读写
- **PageCache**: LRU 热点页缓存

### 3.4 目录管理

- **DatabaseDirectory**: 数据库级别管理（创建/打开/列举/删除）
- **TableFileManager**: 表文件路径管理（文件命名规则/路径生成）

### 3.5 并发与锁管理

引擎内部统一使用 MVCC 实现多读多写（行级不阻塞），嵌入模式与服务器模式**共享同一套引擎代码**。表级文件锁仅是嵌入模式下的跨进程附加协调层。

| 模式 | 引擎并发 | 跨进程锁 | 说明 |
|------|---------|---------|------|
| **嵌入模式** | MVCC 多读多写（进程内） | 表级文件锁 (`{TableName}.lock`)，跨进程单写多读 | 进程内 MVCC 提供行级不阻塞；跨进程通过文件锁协调写入，不同表互不影响 |
| **服务器模式** | MVCC 多读多写 | 无（单进程） | 所有请求通过 SqlEngine 统一调度，MVCC 保证行级读写不阻塞 |
| **只读模式** | 只读 | 无锁 | 不创建 `.wal` 和 `.lock`，多进程可同时只读打开 |

**与 SQLite 对比**：SQLite 写入时锁整个数据库文件（库级锁），NovaDb 每表独立文件组让锁粒度缩小到单表，写 `User` 表不影响 `App` 表的读写，并发度显著提升。

## 4. WAL 日志层 (WAL)

### 4.1 WAL 模式

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| Full | 每次写入同步刷盘 | 金融、关键数据 |
| Normal | 异步刷盘 (1s 间隔) | 通用场景（默认） |
| None | 不写 WAL | 测试、只读场景 |

### 4.2 WAL 记录类型

```
BeginTx → UpdatePage → CommitTx/AbortTx → Checkpoint
```

### 4.3 WAL 生命周期

| 阶段 | 行为 |
|------|------|
| **写入** | 事务写入时，WAL 记录追加到 `{TableName}.wal` |
| **Checkpoint** | 后台线程将 WAL 中的脏页刷新到 `.nvd` 文件 |
| **清理** | Checkpoint 完成后，`.wal` 文件删除（或截断为 0） |
| **崩溃恢复** | 启动时发现 `.wal` 存在 → 执行 WalRecovery 重放已提交事务 → 恢复后删除 `.wal` |
| **只读模式** | 不创建 `.wal` 文件，跳过 WAL 初始化 |

### 4.4 崩溃恢复 (`WalRecovery`)

两阶段恢复策略：
1. **第一阶段 (Analysis)**：扫描 WAL，识别已提交事务
2. **第二阶段 (Redo)**：重放已提交事务的页面更新

## 4.5 Binlog 日志层

Binlog 是数据库级别的逻辑日志，与 WAL（物理日志，表级）互补，用于主从复制、增量备份和审计追踪。

### 4.5.1 WAL 与 Binlog 的关系

```
事务提交流程：
  写操作 → WAL 记录（物理页面变更） → Commit
                                         ↓
                               Binlog 记录（逻辑操作）
                                         ↓
                               推送给从节点
```

| 维度 | WAL | Binlog |
|------|-----|--------|
| 层级 | 表级（`{Table}.wal`） | 数据库级（`nova.*.binlog`） |
| 内容 | 物理页面变更 | 逻辑操作（INSERT/UPDATE/DELETE/DDL） |
| 用途 | 崩溃恢复 | 主从复制 + 增量备份 + 审计 |
| 生命周期 | 短暂（checkpoint 后删除） | 持久保留（滚动清理） |
| 写入时机 | 事务写入时 | 事务提交后 |

### 4.5.2 Binlog 记录格式

```
BinlogRecord:
├── LSN (Int64)              # 全局单调递增序号（断点续传标识）
├── Timestamp (Int64)        # 提交时间 (UTC 毫秒)
├── TableName (String)       # 目标表名
├── OpType (Byte)            # 操作类型: 1=Insert, 2=Update, 3=Delete, 4=DDL
├── RowKey (Variant)         # 主键值（Update/Delete 时定位行）
├── PayloadLength (Int32)    # 数据负载长度
└── Payload (Byte[])         # Insert/Update 的行数据 / DDL 的 SQL 文本
```

### 4.5.3 文件管理

- **文件命名**：`nova.{SeqNo:D6}.binlog`（如 `nova.000001.binlog`）
- **滚动策略**：单文件达到阈值（默认 256MB，`BinlogMaxFileSize` 配置）时关闭当前文件，创建下一个序号文件
- **序号永远递增**：不回绕、不重用，保证全局唯一
- **文件内索引**：FileHeader 之后记录 LSN 范围（起始 LSN ~ 结束 LSN），引擎可快速定位 LSN 所在文件

### 4.5.4 自动清理策略

后台清理线程定期检查，三种策略可叠加使用：

| 策略 | 配置项 | 说明 |
|------|--------|------|
| 从节点确认 | 自动 | 所有从节点已确认同步的文件可安全清理 |
| 按天数保留 | `BinlogRetentionDays` | 超过保留天数的旧文件自动删除 |
| 按总大小限制 | `BinlogMaxTotalSize` | 总大小超过上限时从最早文件开始删除 |

**清理规则**：
- 当前正在写入的活跃文件永远不会被清理
- 按天数/总大小清理时不检查从节点同步状态（运维主动决策，引擎尊重）
- 从节点若请求已删除文件中的 LSN，将收到 Binlog 已清理错误，需全量重建

### 4.5.5 手工管理（安全删除机制）

Binlog 文件设计为**可安全手工删除**的独立文件：
- 每个 `.binlog` 文件自包含（FileHeader + 完整记录序列），文件之间无交叉依赖
- 运维人员可直接在文件系统上 `rm` 旧文件释放磁盘空间，引擎不会崩溃
- 引擎启动/运行时扫描数据库目录，按文件名排序重建 Binlog 文件列表，自动感知已删除的文件

**SQL 管理命令**：

| 命令 | 说明 |
|------|------|
| `SHOW BINLOGS` | 列出所有 Binlog 文件（文件名、大小、LSN 范围、创建时间） |
| `SHOW BINLOG STATUS` | 显示当前活跃 Binlog 文件名、当前 LSN、文件大小 |
| `PURGE BINLOGS TO 'nova.000005.binlog'` | 删除序号 5 之前的所有文件 |
| `PURGE BINLOGS BEFORE '2025-07-01'` | 删除指定时间之前的所有文件 |
| `FLUSH BINLOGS` | 立即关闭当前文件并滚动到新文件 |

### 5.1 MVCC 实现

- **隔离级别**: Read Committed
- **行版本**: `RowVersion(CreatedByTx, DeletedByTx, Payload)`
- **可见性规则**:
  - 当前事务创建且未删除 → 可见
  - 其他活跃事务创建 → 不可见
  - 已提交且未被删除 → 可见
  - 已被已提交事务删除 → 不可见

### 5.2 事务生命周期

```
BeginTransaction → Active → Commit/Rollback → Committed/Aborted
```

### 5.3 回滚机制

每个写操作注册 `RollbackAction`，回滚时按逆序执行，确保数据一致性。

## 6. Nova Engine (关系型引擎)

### 6.1 核心组件

- **NovaTable**: 单表引擎，管理 MVCC 行版本
- **TableSchema**: 表架构定义 (列名/类型/约束)
- **SkipList<K,V>**: 内存跳表索引

### 6.2 索引冷热分离

```
┌────────────────────────────┐
│   热索引 (HotIndexManager)  │  ← 最近访问的数据段
│   完整 SkipList 节点        │     内存中完整索引
├────────────────────────────┤
│   冷索引 (ColdIndexDirectory)│  ← 长期未访问
│   稀疏目录 (每 N 行一个锚点) │     仅保留锚点映射
└────────────────────────────┘
```

- **HotIndexManager**: 管理热段，LRU 淘汰策略
- **ColdIndexDirectory**: 冷段稀疏目录，锚点 → 页偏移

### 6.3 CRUD 数据流

```
Insert:  验证 → 序列化 → 写 WAL → 更新 SkipList → 注册回滚
Get:     主键查找 → MVCC 可见性判断 → 反序列化
Update:  查找 → 标记旧版本删除 → 创建新版本 → 写 WAL
Delete:  查找 → 标记版本删除 → 写 WAL
GetAll:  遍历 SkipList → 按 MVCC 过滤可见行
Truncate: 清空主键索引（SkipList.Clear），保留表结构，比逐行 DELETE 更快
```

## 7. Flux Engine (时序引擎)

### 7.1 数据模型

- **统一模型**：时序数据与 MQ 消息采用同一存储结构。**每一条 MQ 消息就是时序表的一行数据**。
- **FluxEntry**: 时序记录 / MQ 消息 (MessageId + Fields)
- **MessageId**: `{timestamp}-{sequence}`，同毫秒内自增，全局单调递增。消息 ID 同时也是时序表的主键。
- **Stream/Topic 名即时序表名**：发送消息 = 向时序表追加一行，消费消息 = 按 MessageId 顺序读取行。

### 7.2 时间分片

按配置粒度（小时/天）自动创建分片文件：`{StreamName}_{yyyyMMdd}.nvd` 或 `{StreamName}_{yyyyMMddHH}.nvd`

### 7.3 MQ 功能 (Stream)

```
Producer → StreamManager → FluxEngine
                ↓
ConsumerGroup → Consumer → Pending → Ack
```

- **StreamManager**: 流管理器，维护消费组
- **ConsumerGroup**: 消费组，记录消费偏移和 Pending 队列
- **可靠性**: At-Least-Once（读取 → Pending → 业务处理 → Ack）

### 7.4 数据保留

- 按时间/文件大小自动删除旧分片
- TTL 配置通过 `DbOptions.FluxDefaultTtlSeconds`

## 8. KV Store (键值引擎)

### 8.1 实现方式

复用 Nova Engine，通过特殊 Schema 实现 KV 模式。**一个数据库允许创建多个 KV 表**，每个 KV 表用于不同业务场景：

```sql
CREATE TABLE SessionCache (Key STRING(200) PRIMARY KEY, Value BLOB, TTL DATETIME) ENGINE=KV;
CREATE TABLE ConfigCache (Key STRING(200) PRIMARY KEY, Value BLOB, TTL DATETIME) ENGINE=KV;
CREATE TABLE DistLock (Key STRING(200) PRIMARY KEY, Value BLOB, TTL DATETIME) ENGINE=KV;
```

固定 Schema：
```
Key (STRING(200), PK) | Value (BLOB) | TTL (DATETIME, Nullable)
```

### 8.2 API

每个 KV 表实例独立，通过表名区分：
- `Get(key)`: 获取值（惰性检查 TTL）
- `Set(key, value, ttl?)`: 设置值（未指定 TTL 时使用表级默认 TTL）
- `Add(key, value, ttl?)`: 仅当 key 不存在时添加（分布式锁场景）
- `Delete(key)`: 删除键
- `ContainsKey(key)`: 判断键是否存在
- `Inc(key, delta, ttl?)`: 原子递增（计数器场景）
- `GetAll()`: 获取所有有效键值对

### 8.3 TTL 清理

- **读时惰性删除**: Get/ContainsKey 时检查过期
- **后台扫描**: 定期清理过期数据
- **表级默认 TTL**: 每个 KV 表可配置不同的默认 TTL（如 SessionCache 2小时，ConfigCache 24小时）

## 9. SQL 引擎 (Sql)

### 9.1 处理流程

```
SQL 文本 → SqlLexer(词法分析) → SqlParser(语法分析) → SqlAst(AST) → SqlEngine(执行)
```

### 9.2 词法分析器 (`SqlLexer`)

将 SQL 文本分割为 Token 流，支持：
- 关键字识别 (SELECT/INSERT/UPDATE/DELETE/CREATE/DROP 等)
- 标识符、数字/浮点/字符串字面量
- 运算符 (=, !=, <, >, <=, >=, +, -, *, /)
- 参数占位符 (@param)
- 行注释 (--)

### 9.3 语法分析器 (`SqlParser`)

递归下降解析器，支持的 SQL 语句：

| 类别 | 语句 |
|------|------|
| DDL | CREATE/DROP TABLE, CREATE/DROP INDEX, CREATE/DROP DATABASE, ALTER TABLE (ADD/MODIFY/DROP COLUMN, COMMENT), ENGINE |
| DML | INSERT, UPDATE, DELETE |
| 查询 | SELECT, WHERE, ORDER BY, GROUP BY, HAVING, LIMIT, OFFSET |
| JOIN | INNER JOIN, LEFT JOIN, RIGHT JOIN (Nested Loop 执行) |
| 函数 | COUNT, SUM, AVG, MIN, MAX |
| 运算符 | =, !=, <, >, <=, >=, AND, OR, NOT, LIKE, IS NULL, IS NOT NULL |

### 9.4 表达式求值

支持的表达式类型：
- 字面量 (整数/浮点/字符串/布尔/NULL)
- 列引用 (table.column)
- 二元运算 (比较/逻辑/算术)
- 一元运算 (NOT, 负号)
- 聚合函数
- 参数引用

### 9.5 查询执行流程

```
单表查询:
1. 获取所有行 (GetAll)
2. WHERE 过滤
3. GROUP BY 分组
4. HAVING 过滤
5. 聚合函数计算
6. ORDER BY 排序
7. OFFSET/LIMIT 分页
8. 投影 (列选择/别名)

JOIN 查询:
1. 获取左表所有行
2. 依次对每个 JOIN 子句执行 Nested Loop
   - 合并左右行为宽行
   - 按 ON 条件过滤匹配行
   - LEFT JOIN: 无匹配时右侧补 NULL
   - RIGHT JOIN: 无匹配时左侧补 NULL
3. WHERE 过滤
4. ORDER BY 排序
5. OFFSET/LIMIT 分页
6. 投影
```

### 9.6 JOIN 实现

- **AST**: `JoinClause`（类型 + 右表名 + 别名 + ON 条件），`SelectStatement.Joins` 列表
- **解析**: 支持 `[INNER|LEFT|RIGHT] JOIN table [alias] ON condition` 语法
- **执行**: Nested Loop Join，支持多表连续 JOIN，合并列名空间，表前缀列引用解析

## 10. 服务器层 (Server)

### 10.1 架构

基于 NewLife.Remoting.ApiServer 构建，RPC 路由 `Nova/{Method}`：

- **NovaServer**: 服务器封装，管理端口、生命周期和 SqlEngine 实例
- **NovaController**: RPC 控制器，通过静态共享 SqlEngine 处理 SQL 执行、查询和事务管理
- **NovaSession**: 会话管理
- **NovaProtocol**: 二进制协议定义

### 10.2 RPC 接口

| 接口 | 说明 | 实现状态 |
|------|------|----------|
| Nova/Ping | 心跳，返回服务器时间 | ✅ 已实现 |
| Nova/Execute | 执行 SQL（非查询），返回影响行数 | ✅ 已接入 SqlEngine |
| Nova/Query | 查询，返回列名和行数据 | ✅ 已接入 SqlEngine |
| Nova/BeginTransaction | 开始事务，返回事务 ID | ✅ 已接入 TransactionManager |
| Nova/CommitTransaction | 提交事务 | ✅ 已接入 TransactionManager |
| Nova/RollbackTransaction | 回滚事务 | ✅ 已接入 TransactionManager |

### 10.2 协议格式

16 字节固定头部：
```
| 2B Magic(0x4E56) | 1B Version | 1B RequestType |
| 4B SequenceId    | 4B PayloadLength             |
| 1B Status        | 3B Reserved                  |
```

最大载荷: 100MB

### 11.2 客户端工厂 (`NovaClientFactory`)

```csharp
// 通过工厂创建连接
var conn = NovaClientFactory.Instance.CreateConnection();
conn.ConnectionString = "Data Source=/path/to/db";

// 或直接实例化
using var conn = new NovaConnection("Data Source=/path/to/db");

```

工厂在 NETSTANDARD2.1+ / NETCOREAPP 环境下自动注册到 `DbProviderFactories`，注册名为 `NewLife.NovaDb.Client`。

### 11.3 连接字符串 (`NovaConnectionStringBuilder`)

| 属性 | 默认值 | 说明 |
|------|-------|------|
| DataSource | - | 数据源路径（嵌入模式） |
| Server | - | 服务器地址（服务器模式） |
| Port | 3306 | 端口号 |
| Database | - | 数据库名称 |
| UserId | - | 用户名（服务器模式） |
| Password | - | 密码（服务器模式） |
| WalMode | Normal | WAL 模式（嵌入模式，Full/Normal/None） |
| ReadOnly | false | 是否只读模式（嵌入模式） |
| ConnectionTimeout | 15 | 连接超时（秒） |
| CommandTimeout | 30 | 命令超时（秒） |

别名支持：`data source` / `datasource` → DataSource，`server` → Server，`userid` / `user id` / `uid` → UserId，`walmode` / `wal mode` / `wal` → WalMode，`readonly` / `read only` → ReadOnly 等。

### 11.4 连接模式

```
嵌入模式: "Data Source=../data/member" → SqlEngine 直接操作
嵌入只读: "Data Source=../data/member;ReadOnly=true" → SqlEngine 仅读取，禁止写操作
嵌入完整: "Data Source=../data/member;WalMode=Full;ReadOnly=false" → 指定 WAL 模式
服务器模式: "Server=host;Port=3306;Database=member;UserId=root;Password=root" → NovaClient RPC 通信
```

通过 `NovaConnectionStringBuilder.IsEmbedded` 自动识别：当 `DataSource` 非空时为嵌入模式，否则为服务器模式。

### 11.5 NovaClient

远程客户端封装：
- `PingAsync()`: 心跳检测
- `ExecuteAsync(sql)`: 远程 SQL 执行
- `BeginTransactionAsync()`: 远程事务
- `CommitTransactionAsync(txId)`: 提交事务
- `RollbackTransactionAsync(txId)`: 回滚事务

## 12. 集群层 (Cluster)

### 12.1 架构

```
Master ──Binlog 流复制──→ Replica 1
                       └──→ Replica 2
                       └──→ Replica N
```

### 12.2 核心组件

- **ReplicationManager**: 主节点复制管理，Binlog 流发送
- **ReplicaClient**: 从节点客户端，Binlog 流接收与逻辑操作重放
- **NodeInfo**: 节点信息 (NodeId/Endpoint/Role/LSN)

### 12.3 复制策略

- **同步单位**: Binlog 流复制（逻辑日志，非 WAL 物理页）
- **拓扑**: 一主多从
- **断点续传**: 从节点携带已同步 LSN 请求后续 Binlog
- **一致性**: 最终一致 + 可选强一致读（读主）

## 14. 设计决策记录

### D1: 纯托管实现
不引入 native 依赖，保持跨平台一致性。

### D2: MMF 作为主要存储介质
利用操作系统的虚拟内存管理，突破物理内存限制。64 位系统虚拟地址空间 128TB，10G/100G 数据文件 OS 按需分页映射无压力。

### D3: SkipList 而非 B+ 树
SkipList 实现简单、并发友好，适合内存索引场景。

### D4: Read Committed 隔离级别
平衡一致性与并发性能，满足大多数业务场景。

### D5: 表级 WAL
每个表独立 WAL，减少并发竞争，简化恢复流程。写入时产生 `.wal`，checkpoint 后删除；只读模式不创建 `.wal`。

### D6: 冷热分离索引
1000 万行表仅查热点时内存 < 20MB，冷数据自动卸载至 MMF。

### D7: SQL 子集策略
覆盖 ~60% 的 SQL92 标准，聚焦业务常用场景，不追求完整性。

### D8: Nova Engine 不分片
Nova Engine 每表单个 `.nvd` 文件，不做自动数据分片。MMF 可高效处理超大文件，分片带来的路由/边界管理复杂度远大于收益。如需逻辑分表由 XCode 应用层完成。仅 Flux Engine 保留按时间分片。

### D9: 表级并发控制
嵌入模式使用表级文件锁（`{TableName}.lock`），一写多读，不同表互不影响。远优于 SQLite 的库级锁。服务器模式单进程运行，MVCC 提供行级并发。

### D10: nova.db 二进制元数据
`nova.db` 存储数据库完整元数据（表/列/索引定义），二进制格式，程序可读即可。取代 `_sys_tables` 系统表概念，DDL 操作时原子重写保证崩溃安全。

### D11: 嵌入式与服务器存储统一
嵌入模式和服务器模式使用完全相同的文件布局。嵌入模式的数据库目录可直接拷贝到服务器模式使用，反之亦然。

### D12: KV 多表
允许一个数据库创建多个 KV 表（`ENGINE=KV`），每个 KV 表用于不同业务场景，可配置独立的默认 TTL。

### D13: MQ 消息即时序行
Flux Engine 中，MQ 消息就是时序表的一行数据（MessageId + Fields）。Stream/Topic 名即时序表名，统一存储模型简化设计。

### D14: Binlog 独立于 WAL
WAL 是表级物理日志（崩溃恢复），Binlog 是数据库级逻辑日志（主从复制/增量备份/审计）。两者职责正交、互不依赖。WAL checkpoint 后删除不影响 Binlog；Binlog 手工删除不影响 WAL 崩溃恢复。

### D15: Binlog 文件可安全手工删除
每个 `.binlog` 文件自包含（FileHeader + 完整记录序列），文件之间无交叉依赖、无全局索引文件。引擎启动时按文件名排序重建列表，自动感知已删除的文件。运维人员可直接 `rm` 旧文件释放磁盘空间，引擎不崩溃。唯一代价是未同步该文件的从节点需全量重建。该设计借鉴 MySQL Binlog 的运维实践并做了改进——MySQL 官方不建议直接删文件（因为有 binlog.index 索引文件需要同步），NovaDb 去掉了索引文件依赖，使手工删除完全安全。
